## 一、引言

### 1.1 目的

本文档旨在提供 Markdown 编辑器的详细设计，明确各模块的技术实现、接口规范、数据流、核心算法与配置项，指导开发团队进行具体实现。

### 1.2 范围

- **在范围内：** 编辑核心（语法支持、快捷键、选区与历史）、实时预览与双栏布局、文件/文档的本地与云端存储、常用扩展（目录、数学公式、代码高亮、**图表**：Mermaid 流程图/时序图/甘特图/思维导图/ER 图、**ECharts** 数据图表、PlantUML 等）、多人在线协作（实时协同、在线状态、光标/选区同步、权限）、**AI 功能**（续写/改写/润色、翻译、总结、代码解释、智能补全与纠错、**格式化**、**生成目录**）、**批注与评论**（批注锚点、评论线程、回复/@提及/解决、侧栏与行内展示、权限）、**根据用户评论生成待办、生成修改建议、生成新版本**（评论→待办→建议→版本闭环）、多端（Web / 桌面 / 移动）一致性体验；各模块的接口、数据结构、配置与部署考量。
- **不在范围内：** 与具体后端业务系统（如 CMS、博客平台）的深度耦合实现，由单独需求文档约定。

- 开发者（README、技术文档、笔记）
- 知识管理、个人/团队笔记场景

- Markdown 编辑器系统设计（概要）

## 二、核心愿景与目标

**愿景：** 提供「写 Markdown 像写纯文本一样自然、预览像最终文档一样准确」的编辑体验，在保持格式简洁的前提下，兼顾可读性、可扩展性与多端一致。

**核心目标：**

1. **编辑体验优先：** 支持标准 CommonMark/GFM，快捷键与操作符合用户直觉，低延迟输入与撤销/重做。
2. **所见即所得与源码双模式：** 支持纯源码编辑、实时预览、以及（可选）所见即所得（WYSIWYG）混合模式。
3. **扩展友好：** 数学公式、代码高亮、图表（Mermaid 流程图/时序图/甘特图/思维导图/ER 图、ECharts 数据图表、PlantUML 等）、表格、脚注等通过统一扩展机制接入，不污染核心编辑逻辑。
4. **多端一致：** Web、桌面（Electron/Tauri）、移动端共享同一套编辑规范与数据格式，差异化仅体现在交互与布局。
5. **数据可控：** 支持本地文件与可选云端同步，格式以标准 `.md` 为主，避免私有二进制格式锁死。

## 三、系统总体架构

系统采用分层、模块化架构。核心四层自上而下依赖，横切关注点（AI、协作、批注）通过事件总线与各层对接。

```
                          ┌────────────────────────────────┐
                          │          表现层 (UI)           │
                          │  工具栏、侧栏、多标签、设置、   │
                          │  主题、评论面板、待办面板       │
                          └──────────┬───────────────────┘
                                     │ 订阅事件 / 调用接口
                          ┌──────────▼───────────────────┐
                          │         编辑层 (Editor)       │
                          │  编辑器内核、快捷键、选区、    │
                          │  历史、自动配对、多光标        │
                          └──────────┬───────────────────┘
                                     │ contentChange / cursorChange
                          ┌──────────▼───────────────────┐
                          │        文档层 (Document)      │
                          │  Markdown 解析、AST、渲染、   │
                          │  扩展（公式/图表/高亮）、TOC   │
                          └──────────┬───────────────────┘
                                     │ 读写文件 / 云端 API
                          ┌──────────▼───────────────────┐
                          │    存储与同步层 (Storage)     │
                          │  本地文件、IndexedDB、最近、  │
                          │  云端拉取/推送、冲突处理      │
                          └──────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 横切关注点（通过事件总线与各层对接）                         │
  │  ● AI 模块 ─── 调后端 API → 结果写入编辑层                 │
  │  ● 协作模块 ── WebSocket ↔ 编辑层 + Presence → 表现层      │
  │  ● 批注与评论 ─ 评论 API ↔ 表现层 + 编辑层装饰             │
  │  ● 评论驱动 ── AI + 评论 → 待办 / 建议 / 版本              │
  └─────────────────────────────────────────────────────────────┘
```

**各层职责：**

| 层级 | 职责 | 不关心 |
|------|------|--------|
| **表现层** | 布局、主题、工具栏、设置面板、多标签、侧栏（TOC/评论/待办/在线列表） | Markdown 语法、解析、文件 IO |
| **编辑层** | 文本输入、撤销/重做、选区、快捷键、自动配对、多光标、远程光标装饰 | Markdown 语义、渲染、存储 |
| **文档层** | Markdown 解析（CommonMark/GFM）、AST、渲染 HTML、扩展（公式/图表/高亮）、TOC 提取 | 用户交互、文件 IO |
| **存储与同步层** | 本地文件读写、IndexedDB 草稿、最近列表、云端拉取/推送、冲突处理 | 文本编辑、Markdown 语法 |

## 四、模块详细设计

### 4.1 编辑核心模块

- **目标：** 提供低延迟、符合直觉的 Markdown 源码编辑体验，**参照 CommonMark 与 GFM 规范**支持块级与行内语法的编辑与快捷键，不依赖 Markdown 解析结果。
- **技术选型：**
  - **推荐：** CodeMirror 6（模块化、可访问性较好、扩展丰富）。
  - **备选：** Monaco Editor（代码体验强、体积大）、ProseMirror（WYSIWYG/协作场景）。
- **核心功能点详细设计（按 Markdown 规范细化）：**
  - **块级语法支持（CommonMark Block）：**
    - **ATX 标题（#–######）：** 行首 1–6 个 `#` + 空格 + 标题文本；快捷键插入当前行或提升/降级当前标题层级（H1–H6）。
    - **Setext 标题（可选）：** 段落下一行为 `===`（一级）或 `---`（二级）；编辑器可支持插入或由解析层识别。
    - **引用块（>）：** 行首 `>` + 空格；支持多级引用（`>>`）；无选区时在当前行插入 `> `，有选区时对每行加 `> `。
    - **无序列表：** 行首 `-`、`*` 或 `+` + 空格；缩进 2 或 4 空格表示嵌套；快捷键插入 `- ` 或 `* `。
    - **有序列表：** 行首 `数字.` + 空格（如 `1. `）；缩进表示嵌套；快捷键插入 `1. `。
    - **任务列表（GFM）：** 无序列表项内 `[ ]` 或 `[x]` 表示未完成/已完成；快捷键可插入 `- [ ] `。
    - **围栏代码块（```）：** 行首三个反引号 + 可选语言标识 + 换行，结束为三个反引号；快捷键插入 `` ```\n\n``` `` 并聚焦中间；支持语言标识补全（如 `js`、`python`）。
    - **缩进代码块（可选）：** 4 空格或 1 Tab 表示代码块；与段落区分依赖行首空白规则。
    - **表格（GFM）：** `| 列1 | 列2 |`、分隔行 `| --- | --- |`、对齐 `:---` / `---:` / `:---:`；工具栏或快捷键可插入 2×2 表格模板。
    - **主题分隔线：** 行内至少三个 `-`、`*` 或 `_`（可空格），单独成行；快捷键可插入 `---`。
    - **段落：** 由一个或多个非空行组成，与上述块以空行分隔；回车换行、空行插入符合规范。
  - **行内语法支持（CommonMark Inline）：**
    - **强调（*text* 或 _text_）：** 单个 `*`/`_` 包裹；快捷键对选区包裹 `*` 或切换已有强调。
    - **加粗（**text** 或 __text__）：** 双 `*`/`_` 包裹；快捷键 `Ctrl+B` 对选区包裹 `**` 或切换。
    - **行内代码（`code`）：** 反引号包裹，内容中反引号需转义或双反引号；快捷键对选区包裹 `` ` ``。
    - **删除线（GFM，~~text~~）：** 双波浪号包裹；快捷键对选区包裹 `~~`。
    - **链接（[text](url) 或 [text](url "title")）：** 快捷键 `Ctrl+K` 插入 `[]()` 并聚焦 URL；支持从选区生成 `[选区]()`。
    - **图片（![alt](url)）：** 插入 `![]()` 并聚焦 alt 或 url；可选粘贴图片后插入本地/上传 URL。
    - **自动链接（<url>）：** 由解析层识别，编辑器侧可对选中 URL 包裹 `<>`。
  - **扩展语法（可选，由插件声明）：**
    - 脚注（`[^1]` 与 `[^1]: ...`）、自定义容器块（如 `::: tip`）等；编辑器侧仅作纯文本处理，不解析语义。
  - **快捷键与映射：**
    - **块级：** 标题 H1–H6（如 `Ctrl+Alt+1`–`6`）、引用 `Ctrl+Shift+>`、无序列表 `Ctrl+Shift+8`、有序列表 `Ctrl+Shift+7`、围栏代码块 `Ctrl+Shift+`` `、表格（可自定义）、分隔线 `---`。
    - **行内：** 加粗 `Ctrl+B`、斜体 `Ctrl+I`、行内代码（可自定义）、链接 `Ctrl+K`、图片插入（可自定义）。
    - **自定义：** 提供「命令 ID → 快捷键」配置表，支持用户覆盖与多平台（Mac 用 `Meta`，Windows/Linux 用 `Ctrl`）。
  - **选区与格式：**
    - **行内：** 对选区包裹或切换 `**`、`*`、`` ` ``、`~~`；无选区时在光标两侧插入标记并聚焦中间。
    - **块级：** 无选区时在光标行插入 `>`、`- `、`1. `、`` ``` ``、`| col |` 等；有选区时对每行应用块前缀（引用、列表、表格行）。
  - **撤销/重做：**
    - **实现：** 由编辑器内核维护操作历史栈（如 CodeMirror 的 History 插件），单步为一次用户可感知的编辑操作（输入、删除、粘贴等）。
    - **持久化（可选）：** 将历史与文档一起序列化到本地或云端，恢复时回放至上次保存点，需定义历史条目数据结构与版本兼容策略。
  - **自动配对：**
    - **规则：** 输入 `*`、`_`、`` ` ``、`[`、`(`、`$` 等时，在光标后插入对应闭合符并将光标置于中间；围栏代码块 `` ``` `` 可插入三反引号 + 换行 + 三反引号。
    - **配置：** 提供开关（如 `autoPairMarkdown: boolean`），可针对字符级别配置启用/禁用。
  - **多光标与列选区：**
    - **多光标：** 支持 `Ctrl+Alt+Up/Down` 在上/下相同列插入光标，或选中多处后统一编辑。
    - **列选区：** 支持 `Alt+Shift+拖拽` 或块选择模式，便于批量修改多行前缀（如列表符号、表格列）。
  - **查找与替换：**
    - **触发：** `Ctrl+F`（查找）、`Ctrl+H`（替换）打开浮动搜索栏。
    - **能力：** 普通文本匹配 / 正则表达式（可选）；大小写敏感 / 不敏感；全词匹配（可选）。
    - **高亮：** 所有匹配项在编辑区以装饰（高亮背景）标记；当前匹配项以不同色高亮并居中显示。
    - **替换：** 支持「替换当前」与「全部替换」；正则模式下支持捕获组引用（如 `$1`）。
    - **统计：** 搜索栏显示「第 N / 共 M 处匹配」。
  - **图片处理：**
    - **粘贴插入：** 用户从剪贴板粘贴图片时，自动上传（调用配置的图片上传 API 或存入本地）并插入 `![image](url)` 到光标处。
    - **拖拽插入：** 将图片文件拖入编辑区时，同上处理：上传 → 获取 URL → 插入 Markdown 图片语法。
    - **本地引用：** 桌面端可选将图片复制到文档同级 `assets/` 目录并以相对路径引用（如 `![](./assets/img.png)`）。
    - **配置：** `imageUploadEndpoint?: string`（上传 API）、`imageStorageMode: 'upload' | 'base64' | 'local'`。
  - **字数与阅读统计：**
    - **实时统计：** 状态栏显示当前文档字数（中文按字符、英文按单词分别计算）、行数、字符数。
    - **预估阅读时间：** 按可配置的平均阅读速度（如中文 300 字/分钟、英文 200 词/分钟）估算并显示。
    - **选区统计：** 选中文字时状态栏额外显示选中字数与字符数。
  - **折叠与大纲导航：**
    - **代码块折叠：** 围栏代码块可折叠为单行（显示语言标识与首行），点击展开。
    - **标题折叠（可选）：** 按标题层级折叠该标题下的全部内容，便于浏览长文档结构。
    - **面包屑（可选）：** 编辑区顶部或状态栏显示当前光标所在的标题层级路径（如 `## 四 > ### 4.1`），点击可跳转。
  - **与 Markdown 规范对应（编辑核心）：**

    | 规范类型 | CommonMark/GFM 语法 | 编辑核心功能点 |
    |----------|---------------------|----------------|
    | **块级** | ATX 标题 #–###### | 快捷键插入/升降级 H1–H6 |
    | **块级** | 引用 > | 插入 `> `、多行加前缀 |
    | **块级** | 无序列表 - * + | 插入 `- ` / `* `，缩进嵌套 |
    | **块级** | 有序列表 1. 2. | 插入 `1. `，缩进嵌套 |
    | **块级** | 任务列表 - [ ] / [x] | 插入 `- [ ] `（GFM） |
    | **块级** | 围栏代码块（三个反引号） | 插入三反引号+换行+三反引号，语言标识 |
    | **块级** | 表格 \| ... \| | 插入表格模板、列对齐（GFM） |
    | **块级** | 主题分隔线 --- *** | 插入 `---` |
    | **行内** | 强调 * _ | 选区包裹/切换 `*` |
    | **行内** | 加粗 ** __ | 选区包裹/切换 `**` |
    | **行内** | 行内代码（反引号） | 选区包裹反引号 |
    | **行内** | 删除线 ~~ | 选区包裹 `~~`（GFM） |
    | **行内** | 链接 [](url) | 插入 `[]()`、从选区生成 |
    | **行内** | 图片 ![](url) | 插入 `![]()`、粘贴图片 |

- **接口/交互：**
  - **入：** 用户键盘/鼠标输入、外部调用「插入文本」「替换选区」「设置光标」等命令。
  - **出：** 文档内容变更事件（如 `contentChange(doc: string)`）、光标/选区变更事件、撤销栈变更；不直接输出 Markdown AST。
  - **对上层：** 表现层订阅内容与选区事件以驱动预览、保存脏标记；存储层在保存时调用「获取当前文档内容」接口。
- **数据结构（概要）：**
  - **编辑器状态：** 由内核维护（如 CodeMirror 的 `EditorState`），包含文档内容、选区、折叠、装饰等；应用层仅持有引用与监听变更。
  - **快捷键配置：** `Record<string, string>`，键为命令 ID，值为平台归一化后的快捷键字符串（如 `"Mod-b"`）。
- **部署：** 纯前端模块，随应用打包；无服务端依赖。

### 4.2 预览与布局模块

- **目标：** 将当前文档的 Markdown 内容实时、安全地渲染为预览 HTML，并提供多种布局与滚动同步，预览样式与导出一致。
- **技术选型：**
  - **解析/渲染：** markdown-it 或 remark+rehype（见 4.3 文档层）；预览区为独立 DOM 或 iframe。
  - **安全：** DOMPurify 或类似库对渲染结果做 sanitize，再注入预览容器。
- **核心功能点详细设计：**
  - **实时预览：**
    - **触发：** 编辑层内容变更后，经防抖（建议 150–300ms）触发「文档层解析 → 渲染 HTML → sanitize → 写入预览容器」。
    - **节流/防抖：** 防抖延迟可配置（如 `previewDebounceMs`），避免每次按键都重算；大文档可结合虚拟滚动或分块渲染（见 八、性能）。
  - **布局模式：**
    - **枚举：** `editOnly`（仅编辑）、`previewOnly`（仅预览）、`splitHorizontal`（左右分栏）、`splitVertical`（上下分栏）、`sideBySide`（1:1 对照，可选）。
    - **持久化：** 布局模式与分栏比例（如 50%）写入本地配置，下次启动恢复。
  - **滚动同步：**
    - **策略：** 编辑区为主驱动；根据编辑区视口内「可见块」对应的 AST 节点，在预览 DOM 中定位对应块，计算预览目标滚动位置并设置。
    - **块映射：** 解析后建立「源码行号/块 ID → 预览块元素」映射表；编辑区滚动时取视口中心行对应的块，查表得到预览元素，滚动到该元素可见（如 `scrollIntoView` 或设置 `scrollTop`）。
    - **双向（可选）：** 预览区滚动时反向查「预览元素 → 源码行号」，设置编辑区滚动位置，需处理两边滚动条拖动与惯性滚动带来的抖动（可加节流）。
  - **安全渲染：**
    - **流程：** 文档层输出 HTML 字符串 → 经 DOMPurify（或等效）过滤 → 将结果写入预览容器（若为 iframe 则写入 iframe 的 document）。
    - **策略：** 禁止 `script`、`iframe`、`object`、`on*` 事件属性；允许 `a`、`img` 等但 `a` 的 `href` 可限制为 `https?` 或 `#`，或添加 `rel="noopener noreferrer"`；允许样式类名，禁止内联 `javascript:`。
- **接口/交互：**
  - **入：** 文档层提供的「当前 AST 或 HTML」、编辑层提供的「当前滚动位置/可见行」；配置项（主题、字体、是否同步滚动）。
  - **出：** 预览 DOM 更新；滚动同步时写回编辑区滚动位置（若双向）；无直接网络请求。
  - **对上层：** 表现层提供布局切换 UI 与预览容器挂载点；文档层提供渲染结果。
- **数据结构（概要）：**
  - **块映射表：** `Map<blockId, { sourceLineStart, sourceLineEnd, previewElement }>` 或等价结构，用于滚动同步；blockId 可由 AST 节点路径或行号生成。
- **配置项：**
  - `previewDebounceMs: number`
  - `scrollSync: 'none' | 'editToPreview' | 'both'`
  - `layout: LayoutMode`（见上枚举）
- **部署：** 纯前端，无独立服务端。

### 4.3 文档层（Markdown 解析与扩展）

- **目标：** 将 Markdown 原始文本**按 CommonMark/GFM 规范**解析为 AST，并根据模式（预览/导出）渲染为 HTML 或其他格式；支持扩展语法（公式、图表、脚注、TOC 等）通过统一插件机制接入。
- **技术选型：**
  - **解析：** markdown-it（插件多、社区熟）或 remark（unified 生态、AST 统一、易多格式输出）。
  - **公式：** KaTeX（推荐，体积与速度平衡）；MathJax 可选。
  - **代码高亮：** Prism 或 Shiki；Shiki 与 VS Code 主题一致，效果更好。
  - **图表：** Mermaid（前端）；PlantUML 可选（前端 wasm 或后端渲染）。
- **核心功能点详细设计（按 Markdown 规范细化）：**
  - **解析管线与规范对应：**
    - **输入：** 纯文本字符串（UTF-8）；行结束符 LF/CRLF 统一处理。
    - **步骤：** 可选 YAML Front Matter 剥离 → **CommonMark 块级解析**（标题、段落、引用、列表、代码块、表格、分隔线等）→ **行内解析**（强调、加粗、代码、链接、图片、删除线等）→ GFM 扩展（表格、任务列表、删除线、自动链接）→ 扩展插件在 AST 上挂载自定义节点（如 `math`、`mermaid`、`footnote`、`container`）。
    - **块级节点（CommonMark/GFM）：** heading（ATX/Setext）、paragraph、blockquote、list（ordered/unordered/task）、code（fenced/indented）、thematicBreak、table、html（可选禁用）；**行内节点：** emphasis、strong、inlineCode、link、image、delete（GFM）、autolink。
    - **输出：** 统一 AST（如 mdast 或 markdown-it 的 token 流）；供渲染与 TOC 提取使用。
  - **渲染管线：**
    - **预览：** AST → HTML 字符串（含扩展节点渲染）→ 交预览模块 sanitize 后展示。
    - **导出：** AST → HTML/PDF 等（可复用同一套渲染逻辑，仅样式与分页策略不同）。
  - **扩展插件契约：**
    - **解析阶段：** 插件接收「当前 AST 或 token 流」或「原始文本 + 正则/匹配规则」，输出带扩展节点的 AST（或等价结构）。
    - **渲染阶段：** 遇到扩展节点时，由插件返回 HTML 片段（或 React/Vue 组件，再由应用挂载）；插件可异步（如 Mermaid 渲染）。
    - **编辑器增强（可选）：** 插件可注册工具栏按钮、快捷键、自动补全，与编辑层对接。
  - **TOC 提取：**
    - **输入：** AST。
    - **逻辑：** 遍历标题节点（heading），收集层级与文本，生成 `{ level, text, anchor }[]`；anchor 与标题 ID 生成规则一致（如 slugify）。
    - **输出：** 供表现层渲染侧栏或文内目录；预览内标题需设置 `id` 以支持锚点跳转。
  - **公式：**
    - **行内：** 匹配 `$...$`（非贪婪），解析为 math 节点，渲染时调 KaTeX 输出行内 HTML。
    - **块级：** 匹配 `$$...$$`，同理；需处理换行与转义。
  - **代码高亮：**
    - **时机：** 渲染围栏代码块时，根据语言标识调用 Prism/Shiki；无语言时可不高亮或默认纯文本。
    - **主题：** 与编辑器/预览主题一致，配置项如 `codeHighlightTheme: string`。
  - **图表扩展（流程图、时序图、甘特图、思维导图、ER 图、ECharts 等）：**
    - **Mermaid：**
      - **识别：** 围栏代码块语言为 `mermaid`。
      - **常用图类型：** 流程图（flowchart）、时序图（sequenceDiagram）、甘特图（gantt）、**思维导图**（mindmap）、**ER 图**（erDiagram，实体关系图）、类图（classDiagram）、状态图（stateDiagram）、饼图（pie）等，由代码块内首行或语法区分。
      - **渲染：** 将内容交给 Mermaid 引擎，得到 SVG 或图片，嵌入预览 HTML；异步渲染，失败时显示原始代码或错误提示。
    - **ECharts：**
      - **识别：** 围栏代码块语言为 `echarts` 或 `echarts-config`。
      - **内容：** 代码块内为 **JSON 配置**（符合 ECharts option），描述折线图、柱状图、饼图、散点图、雷达图、热力图、地图等数据图表。
      - **渲染：** 前端创建 canvas/div，用 ECharts 实例 `setOption(option)` 渲染；需限制 option 规模与安全（禁止执行函数），可选主题与响应式尺寸。
    - **PlantUML（可选）：**
      - **识别：** 围栏代码块语言为 `plantuml`。
      - **渲染：** 前端 wasm 或后端服务将 PlantUML 源码转为 SVG/图片后嵌入；支持流程图、时序图、甘特图等，与 Mermaid 能力有重叠，可按需选用。
- **接口/交互：**
  - **入：** 原始 Markdown 字符串；渲染模式（preview/export）；扩展插件列表与配置。
  - **出：** AST；HTML 字符串；TOC 列表；可选「块级映射表」供预览滚动同步。
  - **对上层：** 编辑层内容变更后由本模块消费；预览模块消费 HTML 与映射表；表现层消费 TOC。
- **数据结构（概要）：**
  - **AST 节点：** 依所选解析器（mdast/markdown-it tokens）；扩展节点需有统一 type 与 payload。
  - **TOC 项：** `{ level: number, text: string, anchor: string }`。
- **部署：** 纯前端；解析与高亮可放入 Web Worker（见 八、性能）。

### 4.4 文件与文档管理模块

- **目标：** 管理文档的打开、保存、多标签、最近列表与固定项；可选与云端同步服务对接，实现拉取/推送与冲突处理。
- **技术选型：**
  - **Web：** File System Access API（若可用）或上传/下载；LocalStorage/IndexedDB 存草稿、最近列表、配置。
  - **桌面：** Electron/Tauri 直接访问文件系统；用户目录存配置与最近列表。
  - **移动：** 系统文件或 App 沙箱；配置与最近列表存本地存储。
- **核心功能点详细设计：**
  - **本地文件：**
    - **打开：** 用户选择文件或拖拽入；读取为 UTF-8 文本；换行符可统一为 `\n` 或保留原样，可配置（CRLF/LF）。
    - **保存：** 将当前文档内容写回文件；若为「另存为」则弹出路径选择；保存前可做简单校验（如编码是否可表示）。
  - **多标签/多文档：**
    - **模型：** 每个标签对应一个文档实例（路径/ID、内容、脏标记、编辑区状态引用）；标签切换时切换当前文档，编辑区与预览区绑定到当前实例。
    - **未保存提示：** 关闭标签或退出时若脏则提示「保存/放弃/取消」；可支持「全部保存」。
  - **最近与收藏：**
    - **最近列表：** 按「最后打开时间」排序，保留最近 N 条（如 20）；每条含路径、标题（可取文件名）、可选缩略内容。
    - **固定：** 用户可将某文档固定到列表顶部；数据存本地配置。
  - **云端同步（可选）：**
    - **文档列表：** 从服务端拉取当前用户的文档列表（id、title、updatedAt 等）；支持分页与搜索。
    - **拉取/推送：** 打开远程文档时拉取内容；保存时推送到服务端；需认证（如 Token）。
    - **冲突策略：** Last-Write-Wins（时间戳或版本号）；或检测到冲突时提示用户「以本地为准」「以远程为准」或「合并」（合并逻辑可简化或依赖后端）。
- **接口/交互：**
  - **入：** 用户操作（打开/保存/另存为/关闭标签）；编辑层提供的「当前文档内容」与「脏标记」。
  - **出：** 文档列表（本地+远程）；当前文档内容加载完成事件；保存成功/失败回调。
  - **对上层：** 表现层消费文档列表与当前文档元信息；编辑层在「文档切换」时更新内容与历史。
- **数据结构（概要）：**
  - **文档元信息：** `{ id?, path?, title, isDirty, updatedAt?, source: 'local' | 'remote' }`。
  - **最近项：** `{ pathOrId: string, title: string, lastOpenAt: number, pinned?: boolean }[]`。
- **部署：** 本地能力随应用；云端依赖业务后端 API。

### 4.5 导入导出模块

- **目标：** 将当前文档以 Markdown、HTML、PDF、Word/ODT 等格式导出；可选从外部格式导入为 Markdown。
- **技术选型：**
  - **HTML：** 复用文档层渲染管线，包装为完整 HTML 文档（含样式、编码声明）。
  - **PDF：** 基于 HTML 转 PDF（Puppeteer、浏览器打印、或 jsPDF+html2canvas）；或服务端渲染。
  - **Word/ODT：** 可选库如 mammoth（导入）、docx（导出）；或后端服务。
- **核心功能点详细设计：**
  - **导出 Markdown：** 直接取编辑层当前内容，可选带 YAML Front Matter；保存为 `.md` 或下载。
  - **导出 HTML：** 单文件；内联 CSS 或外链样式表；可配置是否内联图片（base64）或保留链接。
  - **导出 PDF：** 调用打印或无头浏览器；页面大小、边距、页眉页脚可配置。
  - **导出 Word/ODT：** 若有前端库则生成 blob 下载；否则请求后端 API 生成并返回文件。
  - **导入：** 支持从 HTML/Word 等解析为 Markdown（质量依赖实现）；导入后写入新标签或当前文档可配置。
- **接口/交互：**
  - **入：** 当前文档内容；导出格式与选项（如 PDF 的页面大小）。
  - **出：** 文件下载或 Blob/URL；错误时提示用户。
- **部署：** 纯前端可完成 MD/HTML/简单 PDF；复杂 PDF 与 Office 可依赖后端。

### 4.6 多人在线协作与多端

- **目标：** 支持多人在线同时编辑同一文档（实时协同）、在线状态与光标/选区可见、评论与批注、文档权限与分享；同一账号下多端共享文档列表与实时内容。
- **技术选型：**
  - **协同算法：**
    - **CRDT（推荐）：** 无中心、最终一致、天然支持离线与多端；典型库 **Yjs**、**Automerge**。Yjs 与 CodeMirror 6 有官方 binding（y-codemirror.next），与 Markdown 纯文本协同契合。
    - **OT（Operational Transformation）：** 需中心服务器做转换与序贯化；典型 **ShareDB**、**Liveblocks**。适合强一致性、需服务端参与权限与审计的场景。
  - **实时通道：** WebSocket 或 WebRTC（Yjs 支持 both）；服务端可选 **y-websocket**、**Hocuspocus**（Yjs）、或自建 WebSocket 房间，按文档 ID 广播操作/状态。
  - **评论与权限：** 与业务后端评论 API、文档权限 API 对接；权限可细化为「仅查看」「可评论」「可编辑」。
- **核心功能点详细设计：**
  - **在线状态（Presence）：**
    - **内容：** 当前文档内在线用户列表：`userId`、`displayName`、`avatar?`、`cursor?`、`selection?`、`lastActiveAt`。
    - **同步：** 通过实时通道广播「加入/离开/心跳/光标/选区」；心跳间隔建议 15–30s，光标/选区节流（如 100ms）后广播，减少流量。
    - **展示：** 侧栏或顶部展示「正在查看本文档的 N 人」；编辑区内为每位协作者渲染**光标与选区**（不同颜色、可带姓名标签），选区高亮半透明。
  - **光标与选区同步：**
    - **数据：** 光标位置用文档内绝对位置（如 CodeMirror 的 `pos` 或 Yjs 的 index）；选区为 `{ from, to }`。需随文档变更做**位置映射**：远端操作应用后，本地光标/选区按 OT 或 CRDT 规则变换，避免错位。
    - **渲染：** 编辑层为「远程用户」维护独立装饰层（如 CodeMirror 的 Decoration）；仅渲染当前可见视口内的远程光标/选区，减少 DOM 数量。
  - **操作同步与冲突解决：**
    - **CRDT（Yjs）：** 本地编辑转换为 Yjs 的 YText 操作，经 WebSocket 广播；远端收到后合并到共享 Y.Doc，binding 自动更新编辑器内容；无需显式冲突解决，最终一致。
    - **OT：** 本地操作序列化为 Op（如 `{ index, insert }` / `{ index, delete, length }`），发送服务端；服务端序贯化、转换后广播；客户端收到远端 Op 后做 OT 变换再应用到本地文档。
  - **文档权限与分享：**
    - **模型：** 文档归属某用户或某空间；可对「用户/链接」授予「仅查看」「可评论」「可编辑」；支持「仅限邀请」与「获得链接者可编辑/可查看」。
    - **校验：** 打开文档时服务端校验权限，返回内容 + 当前用户角色；实时通道连接时携带 Token，服务端校验后允许加入房间或拒绝。
  - **评论与批注：** 与 4.8 批注与评论模块共用数据与 API；协作场景下实时通道可广播「新评论/回复/已解决」以多端即时刷新。锚点、线程、@提及、权限等详见 4.8。
  - **多端一致（同一账号）：**
    - **文档列表与元信息：** 由云端拉取，多端共享；新建/重命名/删除文档即时同步到列表。
    - **内容：** 实时协作场景下，内容已通过 CRDT/OT 多端一致；非实时场景下仍可采用「打开时拉取、保存时推送」+ Last-Write-Wins 或冲突提示。
- **接口/交互：**
  - **入：** 用户加入/离开文档、本地编辑、本地光标/选区、评论操作；实时通道推送的远端操作、Presence、评论事件。
  - **出：** 本地操作上报实时通道；Presence（心跳、光标、选区）上报；评论 CRUD 调用后端 API；文档权限校验请求。
  - **对上层：** 表现层展示在线列表、远程光标/选区、评论侧栏；编辑层消费「远端操作」并更新文档与装饰。
- **数据结构（概要）：**
  - **Presence 项：** `{ userId, displayName, avatar?, cursor?: { from, to }, selection?: { from, to }, color, lastActiveAt }`；`color` 由服务端或客户端按 userId 分配固定色。
  - **评论项：** `{ id, docId, anchor: { line, ch } | { blockId }, content, authorId, createdAt, resolved? }`。
  - **权限：** `role: 'viewer' | 'commenter' | 'editor' | 'owner'`；文档元信息中带 `currentUserRole`。
- **部署：** 实时通道服务（WebSocket 房间/ Hocuspocus）；文档与权限存储、评论 API 由业务后端提供；若采用 Yjs，可选用 y-websocket + 持久化（如 Redis/DB 存 Y.Doc 状态）以支持重连后恢复。

### 4.7 AI 功能模块

- **目标：** 在编辑场景下提供 AI 写作辅助与智能能力，提升写作效率与质量；与编辑层、文档层通过统一接口对接，不侵入核心编辑逻辑。
- **技术选型：**
  - **后端 API（推荐）：** 调用业务侧或第三方大模型 API（如 OpenAI、Claude、国产大模型）；支持流式与非流式；Token 与用量由后端管控。
  - **前端模型（可选）：** 本地或边缘运行小模型（如 WebLLM、Transformers.js）做补全/纠错等轻量能力；适合离线或隐私场景，能力与延迟需权衡。
- **核心功能点详细设计：**
  - **写作辅助：**
    - **续写：** 以当前光标前若干字符或当前段落为上下文，请求「续写下一段」；结果以插入或替换方式写入编辑层，可支持流式打字效果。
    - **改写/润色：** 对当前选区或当前段落请求「改写」「润色」「扩写」「缩写」；结果替换选区或插入到段落下方，用户可接受/拒绝/再编辑。
    - **语气与风格：** 可选「正式/口语/简洁」等风格，作为 prompt 参数传给后端。
  - **翻译：**
    - **选区翻译：** 对当前选区请求翻译为目标语言；结果可插入到选区下方（引用块形式）或弹窗展示，不直接替换原文。
    - **全文翻译（可选）：** 对整篇文档请求翻译；结果可新开标签或导出为译文文件。
  - **总结与提纲：**
    - **选区总结：** 对当前选区（如一段或数段）请求「一句话总结」或「要点列表」；结果在侧栏或弹窗展示。
    - **全文提纲：** 对整篇文档请求生成提纲（Outline）；结果可写入文档顶部或侧栏，供 TOC 参考。
  - **代码相关（可选）：**
    - **代码解释：** 对围栏代码块请求「解释这段代码」；结果在代码块下方折叠块或侧栏展示。
    - **代码补全/生成：** 在代码块内根据上下文请求补全或生成片段；结果插入光标处，用户可接受/修改。
  - **智能补全与纠错：**
    - **补全：** 根据光标前上下文请求「下一个词/句」；以内联建议（Ghost Text）形式展示，用户 Tab 接受；可节流（如 300ms）与取消重复请求。
    - **纠错：** 对当前段落或全文请求「语法/用词纠错」；结果以 diff 或下划线建议形式展示，用户逐条接受/忽略。
  - **格式化：**
    - **选区或全文格式化：** 对当前选区或整篇文档请求「Markdown 格式化」；统一标题层级（如 # ## ###）、列表缩进与符号、代码块围栏与语言标识、空行与换行、表格对齐、引用块与列表嵌套等，符合 CommonMark/GFM 约定。
    - **结果：** 以替换选区或全文方式写入编辑层；可先预览 diff 再接受/拒绝；可选「仅格式化选区」与「全文格式化」。
  - **生成目录：**
    - **基于现有标题：** 对整篇文档请求「生成目录」；解析现有标题（h1–h6），生成 TOC 列表（如 `- [标题](#anchor)`），anchor 与标题 ID 生成规则一致（如 slugify）；可插入到文档顶部指定位置（如 YAML Front Matter 之后）或写入侧栏 TOC 面板。
    - **AI 优化（可选）：** 若文档缺少标题或结构松散，可请求 AI「拟标题并生成目录」：先为段落拟标题再输出 TOC；结果可插入文档（补全标题 + 目录）或仅生成目录供用户确认后插入。
    - **与文档层 TOC 的关系：** 文档层 TOC 为「解析后实时提取」用于侧栏导航；AI 生成目录为「一次性生成 TOC 源码」插入文档，二者可并存（侧栏 TOC 来自解析，文内目录来自 AI 插入）。
  - **与编辑层接口：**
    - **入：** 编辑层提供「当前文档内容」「光标位置」「选区」；用户选择能力类型（续写/改写/翻译等）与可选参数（语言、风格）。
    - **出：** AI 结果（文本或 diff）经统一事件或回调交给编辑层；编辑层执行「插入/替换/装饰」；流式时逐 chunk 更新插入内容。
    - **取消与重试：** 用户取消或切换选区时终止当前请求；失败时支持重试与错误提示。
- **接口/交互：**
  - **入：** 文档内容、选区、能力类型、参数；后端 API 返回的流式或非流式结果。
  - **出：** 调用后端 AI API（或前端模型）；将结果通过「插入/替换/装饰」写入编辑层；表现层展示「加载中/流式/错误」状态。
  - **对上层：** 表现层提供 AI 入口（工具栏、右键菜单、快捷键）；编辑层提供「在指定位置插入/替换」接口供 AI 模块调用。
- **数据结构（概要）：**
  - **请求：** `{ type: 'continue'|'rewrite'|'translate'|'summarize'|'explain'|'complete'|'fix'|'format'|'generateToc', doc, selection?, options? }`；`format` 可选 `scope: 'selection'|'document'`；`generateToc` 可选 `insertAt?: number`、`withHeadings?: boolean`（是否拟标题）。
  - **结果：** `{ text?: string, diff?: Array<{ from, to, replacement }>, stream?: boolean }`；流式时多次返回 chunk。
- **部署：** AI 能力依赖后端 API 或前端模型；API Key 不暴露给前端，由后端代理或业务服务签发 Token；用量与审计由后端统一管理。

### 4.8 批注与评论模块

- **目标：** 支持在文档任意位置添加批注与评论，形成可回复的评论线程，支持 @提及、解决状态、预览/源码双锚点与多端即时更新；可与协作模块配合，也可在单人或「仅评论」权限下独立使用。
- **技术选型：**
  - **锚点与存储：** 批注锚点采用「文档内位置」（行号 + 字符偏移，或块 ID）；文档变更时需**位置映射**（与协作光标类似），或采用稳定块 ID（如基于 AST 块 hash）减少漂移。
  - **后端：** 评论 CRUD、权限校验、可选实时通道广播「新评论/回复/解决」以多端刷新。
- **核心功能点详细设计：**
  - **批注类型与锚点：**
    - **段落批注：** 锚点挂在某一行或某一块（块 ID）；点击行号或块侧边即可在该位置创建批注。
    - **选区批注：** 用户选中一段文字后「添加批注」，锚点记录选区起止（from, to）；文档变更后通过位置映射更新锚点，若选区被删除可标记「已失效」并保留评论内容。
    - **双锚点：** 同一评论在源码侧与预览侧均可定位；源码侧用行号/字符，预览侧用块 ID 或块内偏移；打开评论时根据当前视图（编辑/预览）滚动到对应位置并高亮。
  - **评论线程：**
    - **主评论与回复：** 每条批注对应一个「主评论」；其他用户或本人可「回复」形成线程；数据结构上主评论与回复统一为 Comment，通过 `threadId` 或 `parentId` 关联。
    - **@提及：** 评论内容支持 @userId 或 @displayName；发布时解析提及列表，后端可发送通知（站内信/邮件）；展示时高亮 @ 并可选跳转用户信息。
    - **解决状态：** 主评论可标记为「已解决」；已解决的线程可折叠或单独筛选；仅编辑/所有者或评论创建者可标记解决（可配置）。
  - **展示与交互：**
    - **侧栏：** 评论列表在右侧或左侧面板展示；按文档位置排序或按时间排序；点击某条评论时滚动到对应锚点并高亮（编辑区用装饰、预览区用块高亮）。
    - **行内/块内：** 在编辑区行号旁或预览区块侧显示「评论图标 + 数量」；点击展开该位置下的评论线程。
    - **创建流程：** 用户选中文字或点击行号/块侧 → 弹出评论输入框 → 输入内容（可选 @提及）→ 提交；提交后锚点与评论一并持久化，并可选通过实时通道广播。
  - **权限与可见性：**
    - **谁可评论：** 由文档权限控制（如「可评论」「可编辑」「所有者」可发评论；「仅查看」仅看评论可不发，可配置）。
    - **谁可解决：** 通常为评论创建者、文档编辑/所有者；可配置为「任何人可标记解决」。
    - **谁可见评论：** 拥有该文档查看权限的用户均可见评论列表；私密评论若需支持需单独扩展。
  - **文档变更与锚点：**
    - **位置映射：** 编辑导致行号/字符偏移变化时，批注锚点需随文档变更更新（与协作光标位置映射一致）；若锚点所在行或选区被整体删除，可标记锚点失效并保留评论内容，展示时提示「原位置已变更」。
    - **块 ID：** 若采用稳定块 ID（如基于段落内容的 hash），块内编辑不改变 ID，仅当整段删除时锚点失效，可减少映射计算。
- **接口/交互：**
  - **入：** 用户操作（添加批注、回复、解决、删除）；当前文档内容与选区；后端评论 API 返回的列表/创建/更新结果；可选实时通道的「新评论/回复/解决」事件。
  - **出：** 评论 CRUD 调用后端 API；锚点与评论内容一并提交；可选通过实时通道广播以多端刷新。
  - **对上层：** 表现层展示评论侧栏、行内图标、评论气泡；编辑层提供「在指定位置添加装饰（批注标记）」与「根据锚点滚动定位」；文档层在预览区渲染块级批注标记。
- **数据结构（概要）：**
  - **Comment（扩展）：** `id, docId, threadId?, parentId?, anchor: { line, ch } | { from, to } | { blockId }, content, authorId, authorName?, mentions?: userId[], createdAt, updatedAt?, resolved?, resolvedBy?, resolvedAt?`；主评论无 parentId，回复带 parentId 或 threadId。
  - **锚点：** 段落批注用 `{ line, ch? }` 或 `{ blockId }`；选区批注用 `{ from, to }`（文档内绝对位置）；文档变更后由后端或前端按映射规则更新。
- **部署：** 评论存储与 API 由业务后端提供；若需多端即时刷新，与 4.6 实时通道联动广播评论事件；前端仅做展示与 CRUD 调用。

### 4.9 评论驱动的待办、修改建议与版本

- **目标：** 基于文档内用户评论，由 AI 或规则生成**待办列表**、**修改建议**（可落地的文本替换），并支持**生成新版本**（应用建议后的文档快照或新文件）；打通「评论 → 待办 → 修改建议 → 新版本」的闭环，便于审阅与迭代。
- **技术选型：**
  - **待办生成：** 规则抽取（关键词/模板）或 AI 理解评论内容后输出结构化待办项；可与 4.7 AI 共用后端 API。
  - **修改建议生成：** 以「评论内容 + 锚点处文档片段」为输入，调用 AI 生成「建议替换文本」或 diff；结果与评论、锚点关联。
  - **版本：** 应用建议后得到新内容，可存为文档新版本（版本号/标签）、另存为新文件或导出；版本历史与对比可选由后端或前端维护。
- **核心功能点详细设计：**
  - **根据用户评论生成待办：**
    - **输入：** 当前文档的全部评论（或仅未解决评论）；可选仅选中若干评论。
    - **逻辑：** AI 或规则从评论内容中识别「动作意图」（如「修改第二段」「补充数据来源」「统一术语」），输出结构化待办项：`title`、关联 `commentId`、可选 `anchor`、`status: 'pending'|'done'|'cancelled'`。
    - **输出：** 待办列表（可持久化到后端或本地）；待办与评论双向关联，侧栏可展示「待办面板」，点击待办可定位到对应评论与锚点。
    - **交互：** 用户可「从全部评论生成待办」「从选中评论生成待办」；可编辑/删除/勾选完成待办；完成待办时可可选联动「解决对应评论」。
  - **根据用户评论生成修改建议：**
    - **输入：** 单条评论（或选中多条）+ 评论锚点处的文档片段（根据 anchor 从当前文档截取）。
    - **逻辑：** 调用 AI，以「评论内容 + 原文片段」为上下文，生成「建议替换内容」或 diff（from/to/replacement）；可支持流式。
    - **输出：** 修改建议列表，每项含 `commentId`、`anchor`、`original`、`suggested` 或 `diff`；在编辑区以**内联建议**（装饰/气泡）或侧栏展示，用户可**逐条接受/拒绝**；接受后替换文档对应位置并可选「标记评论已解决」。
    - **交互：** 用户点击评论「生成修改建议」→ 等待 AI 返回 → 展示建议 → 接受/拒绝；支持批量「为所有未解决评论生成建议」。
  - **生成新版本：**
    - **输入：** 当前文档内容 + 当前全部「修改建议」（或用户勾选的部分建议）。
    - **逻辑：** 按锚点顺序对文档应用「接受」后的替换（或直接使用已接受的建议结果），得到**新文档内容**；保存为新版本或新文件。
    - **输出：** 新版本可存为：① 同一文档的版本历史（versionId/标签，如 `v2-from-comments`）；② 另存为新文件（如 `xxx-revised.md`）；③ 导出下载。可选展示「版本对比 diff」便于审阅。
    - **交互：** 用户点击「应用全部建议并生成新版本」→ 确认范围（全部/选中）→ 选择「存为版本」或「另存为文件」→ 完成；可选在版本历史中查看、回滚或对比。
  - **与评论、AI、编辑层的关系：**
    - **评论：** 待办、修改建议均与 `commentId` 关联；解决评论后可同步更新待办状态或建议状态。
    - **AI：** 待办抽取、修改建议生成均调用 4.7 同一后端 AI API；prompt 区分「从评论抽待办」与「根据评论写修改建议」。
    - **编辑层：** 接受修改建议时调用「在指定位置替换」；生成新版本时取「应用建议后的完整内容」写回或另存。
- **接口/交互：**
  - **入：** 评论列表（及选中）、当前文档内容与锚点片段；用户操作（生成待办、生成建议、接受建议、生成新版本）。
  - **出：** 调用 AI API（待办抽取、修改建议生成）；待办 CRUD、建议列表、版本保存调用后端或本地；编辑层执行替换与保存。
  - **对上层：** 表现层提供「待办面板」「建议气泡/侧栏」「生成新版本」入口；评论侧栏可展示「关联待办」「关联建议」。
- **数据结构（概要）：**
  - **待办项：** `{ id, docId, title, commentId?, anchor?, status: 'pending'|'done'|'cancelled', createdAt, completedAt? }`。
  - **修改建议：** `{ id, commentId, docId, anchor, original, suggested | diff, status: 'pending'|'accepted'|'rejected', createdAt }`。
  - **版本：** `{ versionId, docId, content, label?, createdAt, source: 'from-comments'|'manual' }`；可选与「所应用的 suggestionIds」关联。
- **部署：** 待办与建议可存后端（与评论同域）；版本历史若需多端同步则存后端；AI 能力复用 4.7 后端 API。

## 五、系统架构与数据流

### 5.1 总体分层

```
┌─────────────────────────────────────────────────────────────────┐
│  表现层 (UI)                                                     │
│  布局、主题、工具栏、设置面板、多标签                             │
├─────────────────────────────────────────────────────────────────┤
│  编辑层 (Editor)                                                 │
│  编辑器内核、快捷键、选区、历史、自动配对、多光标                 │
├─────────────────────────────────────────────────────────────────┤
│  文档层 (Document)                                               │
│  Markdown 解析、AST、预览渲染、扩展语法、TOC/公式/图表           │
├─────────────────────────────────────────────────────────────────┤
│  存储与同步层 (Storage & Sync)                                   │
│  本地文件、索引、最近列表、云端 API、冲突处理                     │
└─────────────────────────────────────────────────────────────────┘
```

- **表现层：** 不关心 Markdown 语法，只负责把「当前文档」「设置」「预览 HTML」等以合适方式展示。
- **编辑层：** 与「纯文本 + 标记」交互，负责输入、撤销、选区、快捷键；不直接依赖 Markdown 解析结果。
- **文档层：** 将原始文本解析为 AST，再根据模式（预览/导出）渲染为 HTML 或其它格式，并驱动目录、公式等扩展。
- **存储与同步层：** 负责文档的打开、保存、列表、最近记录，以及可选的云端拉取/推送；**多人在线协作时**，与实时通道（WebSocket）协同，负责操作同步、Presence、权限与评论 API。

### 5.2 数据流

#### 5.2.1 核心编辑与预览

1. **用户输入** → 编辑层更新缓冲区文本 → 通知文档层。
2. **文档层** → 解析 Markdown（含扩展）→ 得到 AST → 渲染预览 HTML（或导出格式）。
3. **预览 HTML** → 经安全处理（sanitize）→ 送入预览 iframe/区域。
4. **保存/同步** → 从编辑层取当前文本 → 写入本地文件或调用云端 API。
5. **编辑与预览联动：** 防抖（150–300ms）后触发重新解析与渲染；滚动同步根据编辑区可见块与预览块映射表设置预览滚动位置（可双向）。

#### 5.2.2 多人在线协作（可选）

1. **加入文档：** 拉取文档内容 + 权限 + Presence；建立 WebSocket，加入文档房间。
2. **编辑同步：** 本地编辑 → CRDT/OT 操作 → 实时通道广播 → 远端合并/变换后更新文档与远程光标/选区装饰。
3. **Presence：** 心跳与光标/选区节流上报；收到远端 Presence 后更新在线列表与远程光标/选区渲染。
4. **评论：** 评论 CRUD 走 REST API；可选实时通道广播「新评论/回复/已解决」以多端刷新。

#### 5.2.3 AI 与批注（可选）

1. **AI：** 用户触发能力 → 前端带文档/选区调后端 AI API → 流式或非流式返回 → 结果「插入/替换/装饰」写入编辑层。
2. **批注与评论：** 添加批注/回复/解决 → 调评论 API 持久化并可选广播；文档变更时锚点按位置映射更新或标记失效。
3. **评论→待办→建议→版本：** 从评论生成待办（AI 抽取）→ 根据评论生成修改建议（AI）→ 接受建议后应用并生成新版本（存为版本或另存为文件）。

## 六、技术选型建议

### 6.1 编辑器内核

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| **CodeMirror 6** | 模块化、可访问性较好、扩展丰富，官方 ProseMirror 集成可做 WYSIWYG | 需要深度自定义、插件多的 Web/桌面 |
| **Monaco Editor** | 与 VS Code 同源，代码编辑体验强，体积较大 | 偏「代码式」写作、需要强代码高亮与补全 |
| **ProseMirror** | 文档模型为结构化树，适合 WYSIWYG 与协作 | 以所见即所得或实时协作为主的产品 |
| **textarea + 轻量增强** | 实现简单，功能依赖自行堆叠 | 极简场景、嵌入第三方站点 |

**推荐：** 以「源码为主、预览为辅」的产品优先考虑 **CodeMirror 6**；若产品形态以 WYSIWYG 为主，可选用 **ProseMirror** 并为其定制 Markdown 序列化。

### 6.2 Markdown 解析与渲染

| 层级 | 推荐方案 | 说明 |
|------|----------|------|
| **解析** | **markdown-it** / **remark (unified)** | markdown-it：插件多、社区熟；remark：AST 统一、易做多格式输出（HTML/PDF 等）。 |
| **渲染** | 同解析器配套 | markdown-it 直接出 HTML；remark 配合 rehype 出 HTML，再交给 React/Vue 或裸 DOM。 |
| **公式** | **KaTeX** | 体积与速度平衡好，优先 KaTeX；若有复杂需求可备选 MathJax。 |
| **代码高亮** | **Prism** / **Shiki** | Prism 轻量；Shiki 与 VS Code 主题一致，效果更好，适合桌面/Web 产品。 |
| **图表** | **Mermaid**（流程图、时序图、甘特图、思维导图、ER 图等）+ **ECharts**（数据图表）+ **PlantUML**（可选） | Mermaid：前端渲染 flowchart/sequenceDiagram/gantt/mindmap/erDiagram 等；ECharts：前端 JSON 配置渲染折线/柱状/饼图等；PlantUML：前端 wasm 或后端渲染。 |

### 6.3 存储与运行环境

| 场景 | 建议 |
|------|------|
| **Web** | 本地：LocalStorage/IndexedDB 存草稿、设置、最近列表；文件通过 File System Access API 或上传/下载与后端交互。 |
| **桌面** | Electron/Tauri 直接访问文件系统；配置与最近列表存用户目录。 |
| **移动** | 以「打开/保存」与系统文件或 App 沙箱为主；大文档注意虚拟列表与分片渲染。 |

### 6.4 多端技术栈示例

- **Web：** React/Vue + CodeMirror 6 + markdown-it/remark + KaTeX + Prism/Shiki。
- **桌面：** 同一套 Web 技术栈打包进 Electron 或 Tauri，通过 Native API 做文件对话框、系统菜单等。
- **移动：** 可复用解析与渲染逻辑，编辑器部分采用移动端优化输入（或简化工具栏），布局改为单栏编辑 + 预览切换。

### 6.5 多人在线协作技术栈（可选）

- **协同算法与编辑器绑定：** **Yjs** + **y-codemirror.next**（CodeMirror 6 官方 binding），共享文档为 Y.Doc 中的 YText，与 Markdown 纯文本协同契合；备选 **Automerge**、**ShareDB**（OT）。
- **实时通道：** **y-websocket**（Yjs 官方）、**Hocuspocus**（Yjs 服务端，支持持久化）；或自建 WebSocket 房间，按文档 ID 广播 Yjs 更新 / Presence。
- **后端：** 文档与权限存储、评论 API、Token 校验；若需 Y.Doc 持久化，可选用 Redis/DB 存 Yjs 状态，便于重连恢复与历史版本。

### 6.6 AI 功能技术栈（可选）

- **后端 API：** 调用业务侧或第三方大模型（OpenAI、Claude、国产大模型）；统一代理接口（POST /api/ai/complete 等），流式与非流式；API Key 由后端保管，前端仅传 Token。
- **前端模型（可选）：** WebLLM、Transformers.js 等做轻量补全/纠错；适合离线或隐私场景。
- **与编辑层对接：** AI 结果通过「在指定位置插入/替换」或装饰（diff 建议）写入编辑层；流式时逐 chunk 追加插入内容。

### 6.7 批注与评论技术栈（可选）

- **后端：** 评论 CRUD API（列表、创建、回复、解决、删除）；锚点与评论一并存储；文档变更时由后端或前端按位置映射更新锚点（或采用稳定 blockId）。
- **实时刷新：** 与 4.6 实时通道联动，广播「新评论/回复/解决」以多端即时刷新评论列表。
- **前端：** 评论侧栏、行内/块内批注图标、评论气泡、@提及解析与高亮；编辑区与预览区双锚点定位与高亮。

### 6.8 评论驱动（待办/建议/版本）技术栈（可选）

- **AI：** 复用 4.7 后端 AI API；新增 prompt 类型：「从评论列表抽取待办项」「根据单条评论 + 锚点处原文生成修改建议（替换文本或 diff）」。
- **后端：** 待办 CRUD、修改建议列表与状态、版本历史（版本号/标签、内容快照、来源 `from-comments`）；可与评论、文档同域存储。
- **前端：** 待办面板（与评论关联、勾选完成）、建议侧栏或内联气泡（接受/拒绝）、「生成新版本」入口（存为版本/另存为文件）、版本对比 diff。

## 七、数据结构与接口

以下类型按领域分组：**编辑与文档**（配置、文档元信息、最近项）、**协作**（Presence、评论、角色）、**AI**（请求/结果类型）、**评论驱动**（待办、修改建议、版本）。事件契约与扩展插件契约见 7.3、7.4。

### 7.1 文档在内存中的表示

- **编辑侧：** 纯文本字符串（或由编辑器内核维护的文档模型，如 CodeMirror 的 Doc）。

### 7.4 扩展插件契约

- **解析阶段：** 输入为 Markdown 字符串或 AST，输出为带扩展节点的 AST（如公式节点、Mermaid 节点）；插件需声明 `name`、`parse(txt|ast) => ast`。

## 八、安全与性能

### 8.1 安全

- **预览 XSS：**
  - **实现：** 所有预览 HTML 经 DOMPurify（或等效）过滤后再注入预览容器。
  - **配置要点（DOMPurify）：**
    - `ALLOWED_TAGS`：允许常见排版与媒体标签（如 `h1–h6,p,ul,ol,li,a,img,code,pre,blockquote,table,span,div`），禁止 `script,iframe,object,embed,form,input,button`。
    - `ALLOWED_ATTR`：允许 `href,src,alt,class,id,target,rel` 等；禁止 `on*`、`formaction`、`javascript:`。
    - `ADD_ATTR`：对 `<a>` 强制添加 `rel="noopener noreferrer"`；对 `target="_blank"` 的链接必须配合使用。
  - **策略：** 禁止内联 `javascript:` URL；`data:` 仅允许 `data:image/*`（可选）。
- **链接：** 外部链接一律 `rel="noopener noreferrer"`；可选「首次点击外部链接时弹窗确认」策略。
- **本地文件：** 桌面端打开任意路径文件时，仅将内容作为纯文本读入编辑器与解析管线，不执行任何脚本；预览仅渲染经 sanitize 的 HTML。
- **ECharts 配置安全：** 围栏代码块内的 ECharts option 仅解析为**纯 JSON**，禁止在 option 中传入或执行函数（如 formatter、回调）；渲染前可做 JSON 校验与白名单字段过滤，避免注入或异常 option 导致的问题。

### 8.2 性能

- **阈值与策略：**
  - **大文档行数阈值：** 建议 3000 行（可配置）。超过时：
    - 编辑区：依赖编辑器内核的视口内渲染（CodeMirror/Monaco 均支持）。
    - 预览：可启用「仅渲染视口附近块」或分块渲染，避免一次性生成整篇 DOM。
  - **预览防抖延迟：** 建议 150–300ms（可配置 `previewDebounceMs`）；输入停止后触发解析与渲染。
  - **滚动同步节流：** 编辑区滚动驱动预览滚动时，建议 50–100ms 节流，避免滚动过程中频繁计算与重排。
- **扩展按需加载：** 公式（KaTeX）、Mermaid、ECharts 等仅在文档中检测到对应语法（如 `$`、` ```mermaid `、` ```echarts `）时动态加载对应运行时；首屏可仅加载解析与基础渲染。
- **Worker：**
  - **解析：** 将 Markdown 解析（及扩展插件中的纯计算部分）放入 Web Worker，主线程仅接收 AST 或 HTML 结果并更新 DOM。
  - **高亮：** 代码高亮（Prism/Shiki）可在 Worker 中执行，主线程接收已着色 HTML 片段；需注意 Worker 与主线程间序列化成本。
  - **接口：** `postMessage({ type: 'parse', payload: string })` → `{ type: 'parsed', payload: AST | html }`；错误通过 `type: 'error'` 回传。

### 8.3 错误处理策略

| 场景 | 策略 | 用户反馈 |
|------|------|----------|
| Markdown 解析失败 | 回退至纯文本预览；记录错误日志 | 预览区顶部显示警告条「解析异常，已回退纯文本」 |
| 扩展渲染失败（Mermaid/ECharts/KaTeX） | 在对应块位置显示原始代码 + 错误信息；不影响其他块渲染 | 代码块下方红色提示「渲染失败：{错误摘要}」 |
| 文件保存失败 | 重试一次（500ms 后）；失败则保留脏标记并提示用户 | Toast 提示「保存失败，请检查磁盘空间或权限」；脏标记保留 |
| 云端同步冲突 | 检测到版本不一致时弹窗，提供「以本地为准」「以远程为准」「手动合并」选项 | 模态弹窗含 diff 预览 |
| AI 请求超时/失败 | 自动重试一次（指数退避）；仍失败则取消 loading 状态 | Toast 提示「AI 请求失败，请稍后重试」 |
| AI 流式断开 | 保留已接收内容，提示用户「部分结果已写入，可手动继续」 | 编辑区保留已插入内容 + 警告提示 |
| WebSocket 断连（协作） | 自动重连（指数退避，最多 5 次）；重连后同步 Y.Doc 差量 | 顶部横幅「连接已断开，正在重连…」→「已重新连接」 |
| 评论 API 失败 | 乐观更新回滚；Toast 提示 | 撤销本地新增/修改 + 错误提示 |
| 图片上传失败 | 提示用户并在光标处插入 `![上传失败]()`；可选存为 base64 临时保留 | Toast 提示「图片上传失败」 |

### 8.4 无障碍（Accessibility）

- **编辑器可访问性：**
  - CodeMirror 6 原生支持 ARIA 属性与屏幕阅读器；确保启用 `accessibilityLabel` 与 `role="textbox"`。
  - 所有工具栏按钮需设置 `aria-label` 与 `title`，便于屏幕阅读器与键盘导航。
  - 快捷键操作均可通过工具栏按钮替代，确保仅用键盘可完成所有操作。
- **预览区：**
  - 预览 HTML 中标题设置 `id` 与层级语义标签（`<h1>`–`<h6>`），便于辅助工具导航。
  - 图片 `<img>` 使用 Markdown 中的 `alt` 文本作为 `alt` 属性。
  - 代码块使用 `<pre><code>` 语义标签并设置 `aria-label="代码块"` 或对应语言名。
- **色彩对比度：**
  - 主题配色（浅色/深色）符合 WCAG 2.1 AA 标准（文本与背景对比度 ≥ 4.5:1、大文本 ≥ 3:1）。
  - 语法高亮配色需确保不同 token 可辨识，不仅依赖色相（可选加粗/斜体区分）。
- **键盘导航：**
  - 所有面板（侧栏、评论、待办、设置）可通过 Tab / Shift+Tab 切换焦点。
  - 弹窗/模态框聚焦后 Escape 关闭，焦点回到触发元素。
  - 评论列表与待办面板支持上/下方向键逐项移动。

### 8.5 国际化（i18n）

- **文案外置：** 所有 UI 文案（按钮、提示、菜单、错误信息）通过 i18n key 引用，不硬编码中文或英文字符串；推荐使用 `i18next` 或 `vue-i18n` 等成熟方案。
- **默认语言：** 跟随浏览器/系统语言偏好，可在设置中手动切换。
- **RTL 支持（可选）：** 若需支持阿拉伯语/希伯来语等 RTL 语言，编辑区与预览区需具备 `dir="rtl"` 切换能力，布局镜像。
- **日期/数字格式：** 时间戳显示（如评论时间、版本时间）使用 `Intl.DateTimeFormat` 按用户 locale 格式化。

## 十一、部署与运行环境

### 11.1 Web

- **存储路径与用途：**
  - **LocalStorage：** 配置（`EditorSettings`）、布局与主题、最近列表（`RecentItem[]`）；单 key 如 `md-editor-settings`、`md-editor-recent`；注意单域 5MB 限制。
  - **IndexedDB：** 草稿（未保存文档内容）、大块历史或离线缓存；库可选 idb、Dexie；按 `docId` 或 `path` 建索引。
- **文件访问：** File System Access API（Chrome/Edge）可直接读写在用户选择的目录；其余浏览器通过「选择文件」上传、保存时触发下载（文件名由用户指定）。
- **环境变量：** 可选 `VITE_API_BASE` 等指向云端文档 API；构建时注入。

### 11.2 桌面（Electron / Tauri）

- **配置与最近列表：** 存用户目录，如 Electron 的 `app.getPath('userData')`，路径示例：`userData/settings.json`、`userData/recent.json`。
- **文件系统：** 直接使用 Node/系统 API 读写用户选择的路径；保存时若文件已被外部修改，可提示「覆盖/重新加载」。
- **原生能力：** 文件对话框、系统菜单、快捷键（应用级）、托盘；Tauri 需在配置中声明所需权限。

### 11.3 移动

- **存储：** 配置与最近列表存本地存储（LocalStorage 或平台等效）；文档内容以「打开/保存」与系统文件或 App 沙箱交互。
- **布局：** 单栏为主；编辑与预览通过 Tab 或底部切换，避免左右分栏占用空间。
- **性能：** 大文档启用虚拟列表与预览分块；解析与高亮建议放入 Worker，避免输入卡顿。

### 11.4 配置持久化约定

- **格式：** JSON；写入前可做 schema 校验与默认值合并。
- **版本：** 配置对象可含 `version: number`，升级时做迁移（如重命名 key、拆分存储）。
- **同步：** 多端间配置不同步时，以「当前端」为准；若需同步，由业务后端提供配置 API。

## 十二、开发与测试建议

### 12.1 分层与依赖

- **依赖方向：** 表现层 → 编辑层、文档层、存储层；编辑层、文档层、存储层之间通过事件或接口通信，避免表现层直接操作编辑器内核或解析器内部。
- **可测试性：** 编辑层对外暴露「获取内容/设置内容/执行命令」接口，便于单测；文档层「解析 → AST → 渲染」可纯函数化，便于单元测试与快照。

### 12.2 单元与集成测试

- **编辑层：** 用例包括「插入文本后内容正确」「快捷键触发后选区与内容符合预期」「撤销/重做后状态一致」「查找与替换结果正确」「图片粘贴/拖拽后插入语法正确」「字数统计准确」；可基于 JSDOM 或真实编辑器实例。
- **文档层：** 用例包括「给定 Markdown 输出预期 HTML」「TOC 提取结果与标题一致」「扩展语法（公式、Mermaid、ECharts）被正确识别与渲染占位」；快照测试可用于 HTML 输出。
- **预览与滚动同步：** 集成测试中模拟编辑区滚动，校验预览区滚动位置与块映射一致；可 mock 块映射表。

### 12.3 端到端与兼容性

- **E2E：** 关键路径覆盖「打开文件 → 编辑 → 预览更新 → 保存 → 再次打开内容一致」；多标签「新建/切换/关闭未保存提示」。
- **兼容性：** Web 端需在目标浏览器（Chrome/Firefox/Safari/Edge）验证 File API、Worker、可选 File System Access API；桌面端在 Windows/macOS/Linux 各验证文件路径与快捷键。

### 12.4 性能与监控

- **指标：** 首屏可编辑时间、输入到预览更新延迟（P95）、大文档（如 3000 行）下的滚动帧率与内存占用。
- **监控：** 生产环境可上报解析错误、扩展渲染失败（如 Mermaid、ECharts 报错）、保存失败；不上报文档内容本身。

### 12.5 可选模块测试（协作 / AI / 批注 / 评论驱动）

| 模块 | 测试要点 | Mock/依赖 |
|------|----------|-----------|
| **协作** | 多人同时编辑最终一致；Presence 与远程光标/选区正确；权限（仅查看不可编辑、可评论者可发评论）；断线重连恢复 | 实时通道、后端权限 API |
| **AI** | 续写/改写结果正确插入或替换；流式逐 chunk 更新；取消与错误恢复；补全 Tab 接受后内容正确 | 后端 AI API |
| **批注与评论** | 添加批注后锚点与评论持久化；回复、@提及、解决状态正确；文档变更后锚点映射或失效提示；侧栏点击定位 | 评论 API |
| **评论驱动** | 从评论生成待办列表与关联正确；根据评论生成建议后展示、接受/拒绝后文档更新；应用建议生成新版本后内容正确、版本历史可查 | AI API、待办/版本 API |

| **Ghost Text** | 智能补全的行内灰色建议文本，用户按 Tab 接受 |

| **Block Map** | 源码行号/块 ID 与预览 DOM 元素的映射表，用于滚动同步 |
| **Front Matter** | Markdown 文件开头的 YAML 元数据块（以 `---` 界定） |
| **Slugify** | 将标题文本转换为 URL 安全的锚点 ID 的算法（如「Hello World」→ `hello-world`） |
| **MVP** | Minimum Viable Product（最小可行产品）— 满足核心需求的最小功能集 |
| **SSE** | Server-Sent Events — 服务端向客户端推送流式数据的 HTTP 协议 |
| **WebSocket** | 全双工持久连接协议，用于实时双向通信 |
| **Web Worker** | 浏览器后台线程，用于将计算密集任务（解析、高亮）移出主线程 |
| **IndexedDB** | 浏览器端大容量结构化存储，适合草稿、历史等数据 |
| **ARIA** | Accessible Rich Internet Applications — Web 无障碍属性规范 |
| **WCAG** | Web Content Accessibility Guidelines — Web 内容无障碍指南 |
