# 安全检查 Skill

这个 Skill 用于指导 AI 或工程师系统性评估代码和架构中的安全风险，确保安全防护从设计、开发到上线都得到有效覆盖。

## 适用场景
- 代码提交或 PR 审查时的安全检查。
- 新功能上线前的安全风险评估。
- 业务变更、依赖升级后的安全复核。
- 线上安全事件或漏洞修复后的回归检验。
- 应用架构、接口和权限设计评审。

## 安全检查目标
- **发现漏洞**: 识别注入、越权、敏感信息泄露、后门、开放端口等风险。
- **评估影响**: 明确风险范围、影响对象和可能造成的后果。
- **验证防护**: 检查认证、授权、输入校验、输出编码、加密和配置等防护措施。
- **降低风险**: 提出可执行的修复建议，并避免造成系统可用性或性能大幅下降。
- **提升可审计性**: 确保异常、日志、监控和报警为安全事件排查提供支持。

## 检查原则
- **全面覆盖**: 不只看代码，还要看配置、依赖、部署和接口边界。
- **风险导向**: 优先查看高风险入口、高权限路径和敏感数据处理流程。
- **可复现**: 有条件时验证漏洞是否可复现，而不是只停留在理论判断。
- **最小权限**: 任何访问、执行、权限判断都应遵循最小权限原则。
- **按需输出**: 给出明确修复方案和验证方式，而不是笼统建议。

## 安全检查清单

### 1. 认证与授权
- 是否存在未鉴权访问的接口或页面？
- 是否存在纵向越权、横向越权或租户隔离失效？
- 是否对敏感操作、管理操作和数据查询做了额外权限检查？
- 是否正确区分身份认证和业务授权？
- 是否存在权限判定逻辑绕过风险？

### 2. 输入与输出保护
- 是否对所有外部输入进行类型、边界、长度、格式校验？
- 是否存在 SQL 注入、命令注入、模板注入、路径遍历、XML 注入等风险？
- 是否对前端输出进行了 HTML/JS/URL 编码防止 XSS？
- 是否避免在日志、错误消息、响应中泄露敏感信息？

### 3. 数据安全与隐私
- 是否对敏感数据（密码、Token、身份证、银行卡、隐私字段）做了加密、脱敏、访问控制？
- 是否避免在客户端、日志、缓存、URL 中泄露敏感数据？
- 是否遵循最小必要原则，避免无谓的数据收集和传输？
- 是否存在不安全的会话管理、Cookie 设置、Token 存储？

### 4. 依赖与环境
- 是否使用了最新稳定的依赖版本，是否关注已知漏洞？
- 是否存在本地或远程依赖加载风险、依赖注入漏洞？
- 是否避免将密钥、凭证、配置写入仓库或日志？
- 是否有安全相关的环境变量、配置和部署隔离？

### 5. 网络与接口安全
- 是否对外部请求设置了超时、重试、重定向、TLS、证书校验？
- 是否限制了敏感 API 的访问来源、频率和调用方式？
- 是否避免了开放端口、开放管理接口、错误的 CORS 或 CSP 配置？
- 是否对第三方回调、Webhook、文件上传做了签名校验、来源校验和白名单？

### 6. 日志与可观测性
- 是否记录了安全关键事件、失败原因、身份上下文？
- 是否避免日志中泄露敏感字段、Token、密码、内部路径？
- 是否有安全监控、告警、审计日志和事件追踪能力？
- 是否能有效定位异常请求来源和攻击面？

## 验证与建议
- 是否补充了安全测试、攻击面检测或自动化扫描？
- 是否提供修复后验证步骤和回归检查点？
- 是否评估了修复方案的稳定性和兼容性？
- 是否提出了风险缓解措施而不是单纯的“修复”要求？

## 推荐输出结构
- **问题描述**: 风险点、漏洞类型、发现位置。
- **影响分析**: 可能造成的安全后果及受影响对象。
- **修复建议**: 具体修改方案、优先级、验证步骤。
- **验证结果**: 复现方式、测试覆盖、修复后验证要点。
- **安全监测**: 建议新增的日志、告警、审计与回归验证。
