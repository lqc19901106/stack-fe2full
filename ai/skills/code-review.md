# Code Review Skill

这个 Skill 用于指导 AI 或工程师进行代码审查，目标不是挑格式问题，而是发现真实风险：功能缺陷、边界遗漏、安全漏洞、性能退化、可维护性问题、测试缺口和上线风险。

## 适用场景
- Pull Request / Merge Request 审查。
- 功能开发完成后的代码检查。
- 重构前后的风险评估。
- Bug 修复是否完整的验证。
- 安全、性能、稳定性专项审查。
- AI 生成代码的质量检查。
- 上线前核心链路复核。

## Code Review 目标
- **正确性**: 代码是否满足需求，是否存在明显逻辑错误。
- **完整性**: 是否覆盖正常流程、异常流程、边界场景和权限场景。
- **安全性**: 是否引入越权、注入、XSS、敏感信息泄露等风险。
- **可维护性**: 代码结构是否清晰，命名是否准确，职责是否合理。
- **可测试性**: 核心逻辑是否容易测试，是否有足够测试覆盖。
- **性能稳定性**: 是否存在重复请求、低效查询、资源泄露、阻塞操作。
- **兼容性**: 是否破坏已有接口、数据结构、配置或用户行为。
- **可观测性**: 是否有必要的日志、错误信息、指标或排障线索。

## Review 前需要收集的信息
- **需求背景**: 这次改动要解决什么问题。
- **改动范围**: 涉及哪些页面、接口、服务、数据库、配置和部署。
- **关键路径**: 用户或系统最核心的执行流程是什么。
- **风险点**: 是否涉及权限、支付、数据迁移、删除、外部回调、异步任务。
- **测试情况**: 作者跑了哪些测试，是否有截图、日志、接口结果或压测数据。
- **兼容要求**: 是否需要兼容历史数据、老接口、旧客户端或线上配置。

## Review 输出原则
- **问题优先**: 先列真实风险，再给建议，不要先写大段总结。
- **按严重程度排序**: Critical、High、Medium、Low。
- **具体到代码位置**: 指出文件、函数、逻辑块或变更点。
- **说明影响**: 为什么这是问题，会造成什么后果。
- **给出修复方向**: 不只是说“这里有问题”，要说明怎么改。
- **区分阻塞与建议**: 哪些必须修，哪些是优化建议。
- **避免无价值评论**: 不要把个人风格偏好包装成阻塞问题。

## 严重程度定义
- **Critical**: 会导致数据丢失、严重安全漏洞、核心链路不可用、生产事故，必须阻塞合并。
- **High**: 会导致功能错误、越权、明显性能问题、严重兼容问题，通常需要合并前修复。
- **Medium**: 边界场景、可维护性、错误处理或测试缺口，建议合并前修复。
- **Low**: 命名、注释、轻微重复、非关键优化，不应阻塞紧急修复。

## 通用 Review 清单

### 1. 需求与行为
- 代码是否真正实现了需求，而不是只覆盖 happy path。
- 是否遗漏权限、状态、角色、租户、时间、地区、语言等业务条件。
- 是否处理重复提交、重复请求、刷新页面、回退页面和并发操作。
- 是否改变了已有行为，且这种改变是否被明确接受。
- 是否存在隐含假设，例如数据一定存在、接口一定成功、用户一定有权限。

### 2. 代码结构
- 模块职责是否清晰，是否把过多逻辑堆在一个函数或组件里。
- 命名是否准确表达业务含义。
- 是否存在重复逻辑，可以通过已有工具函数或局部抽象减少复杂度。
- 是否引入过度抽象，让简单需求变得难理解。
- 是否遵循项目已有目录结构、代码风格和依赖方式。

### 3. 错误处理
- 外部请求、数据库操作、文件操作、异步任务是否处理失败。
- 错误信息是否对用户可理解，对开发者可排查。
- 是否错误吞掉异常，导致失败静默发生。
- 是否把内部错误、堆栈、SQL、密钥或敏感路径暴露给用户。
- 是否有重试、降级、回滚或补偿机制。

### 4. 安全
- 是否存在未鉴权接口、越权访问、跨租户数据泄露。
- 是否校验所有外部输入，包括 body、query、params、header、cookie、文件和 webhook。
- 是否存在 SQL 注入、命令注入、路径穿越、模板注入。
- 前端是否存在 XSS、危险 HTML 渲染、敏感 Token 暴露。
- 是否把密钥、Token、密码、证书写入代码、配置、日志或测试数据。
- 文件上传是否限制类型、大小、数量、存储权限和访问权限。

### 5. 数据与数据库
- 表结构、字段类型、默认值、索引、唯一约束是否合理。
- 查询是否可能全表扫描、N+1 查询或返回过多数据。
- 写操作是否有事务边界，失败时是否会产生脏数据。
- 是否考虑并发更新、幂等、重复消费和状态冲突。
- 数据迁移是否向前兼容，是否有回滚和历史数据处理方案。
- 删除操作是否误删、越权删除或破坏引用关系。

### 6. API 与服务
- 接口路径、方法、状态码、错误码和响应结构是否符合项目规范。
- 分页、排序、过滤、幂等、限流、鉴权是否完整。
- 是否对第三方调用设置超时、重试、熔断和签名校验。
- 服务间调用是否传递 traceId、用户上下文和必要权限上下文。
- 是否破坏接口兼容性，尤其是字段删除、类型变化、枚举变化。

### 7. 前端页面
- 是否处理 loading、empty、error、success 状态。
- 表单是否有默认值、校验、提交中状态、重复提交防护。
- 列表是否处理分页、筛选、排序、重置、权限按钮和空数据。
- 危险操作是否有二次确认和明确反馈。
- 是否存在不必要重渲染、重复请求、内存泄漏或竞态问题。
- 是否考虑响应式、可访问性、键盘操作和基础语义化。

### 8. 后端服务
- Controller/Handler、Service、Repository、DTO、Domain Model 职责是否清晰。
- 业务规则是否集中，是否散落在多个入口造成不一致。
- 是否正确处理认证、权限、事务、幂等、并发和状态机。
- 后台任务是否可追踪、可重试、可取消、可观察。
- 消息消费是否处理重复消息、乱序、失败重试和死信队列。

### 9. 性能与资源
- 是否引入重复计算、重复请求、低效循环、大对象复制。
- 是否可能造成内存泄漏、连接泄漏、文件句柄泄漏。
- 是否有未限制的并发、无界队列、无限重试。
- 是否对大列表、大文件、大查询、大响应做限制。
- 缓存是否有 TTL、失效策略和一致性考虑。

### 10. 测试
- 是否新增或更新单元测试、集成测试、接口测试或 E2E 测试。
- 测试是否覆盖正常流程、失败流程、边界条件和权限场景。
- Bug 修复是否包含能复现旧问题的回归测试。
- 数据迁移、异步任务、第三方回调是否有可验证测试。
- 是否只测实现细节，缺少对用户行为或业务结果的验证。

### 11. 可观测性
- 关键失败是否有日志，日志是否包含 traceId、业务 id、错误原因。
- 是否避免日志泄露敏感数据。
- 核心链路是否有指标，例如请求耗时、错误率、任务失败率。
- 是否方便线上排查，而不是只能本地复现。
- 告警和监控是否需要同步调整。

### 12. 部署与运维
- 是否新增环境变量、配置项、依赖服务、端口、权限或证书。
- 是否更新 Dockerfile、CI/CD、Kubernetes、部署文档。
- 是否有数据库迁移、脚本执行顺序、回滚方案。
- 是否需要灰度、特性开关或兼容发布。
- 是否有上线后验证步骤和失败回滚步骤。

## 不同变更类型的重点

### 新功能
- 需求是否完整实现。
- 权限、异常、边界、空状态是否覆盖。
- 新增接口、表结构和配置是否合理。
- 是否有测试和文档更新。

### Bug 修复
- 是否真正修复根因，而不是只修表面现象。
- 是否可能引入回归。
- 是否补充回归测试。
- 是否需要修复同类问题的其他位置。

### 重构
- 外部行为是否保持一致。
- 是否有足够测试保证重构安全。
- 是否真的降低复杂度，而不是换一种复杂方式。
- 是否混入功能变更。

### 性能优化
- 是否有性能数据支撑。
- 优化是否改变业务结果。
- 是否引入缓存一致性、并发或内存风险。
- 是否有压测或对比指标。

### 安全修复
- 是否覆盖所有入口和同类问题。
- 是否影响正常用户流程。
- 是否补充安全测试或权限测试。
- 是否需要密钥轮换、日志清理或历史数据处理。

### 数据库变更
- DDL 是否兼容线上数据。
- 大表变更是否会锁表或影响性能。
- 是否有迁移、回滚、备份和验证步骤。
- 应用发布和数据库发布顺序是否安全。

## Code Review 注意事项
- 不要把所有建议都当成阻塞项。
- 不要在不了解业务上下文时武断要求大改架构。
- 不要只说“建议优化”，要说明为什么、怎么优化。
- 不要忽略测试、部署、配置和数据迁移。
- 不要只看新增代码，也要看它如何影响旧逻辑。
- 不要只检查语法，要关注用户行为和生产风险。

## Code Review 能力提升建议
- 多读线上事故复盘，理解常见缺陷如何演变成事故。
- 熟悉项目架构、业务规则和历史约束。
- 建立自己的检查清单，并按项目类型不断补充。
- 学会阅读 diff，也要学会跳出 diff 看调用链和数据流。
- 对安全、数据库、并发、缓存、异步任务保持敏感。
- Review 后关注问题是否真的被修复，形成闭环。
