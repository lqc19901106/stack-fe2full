# 自动化测试脚本编写 Skill

这个 Skill 用于指导 AI 或工程师为前端、后端、接口、端到端流程和 CI/CD 编写自动化测试脚本，目标是用可维护、可重复执行的测试保障核心业务质量，而不是为了覆盖率堆砌脆弱测试。

## 适用场景
- 新功能开发完成后，需要补充单元测试、集成测试、接口测试或 E2E 测试。
- Bug 修复后，需要编写能复现旧问题的回归测试。
- 重构前后，需要用测试锁定外部行为，降低改动风险。
- 上线前，需要为核心链路、权限、异常流程和边界条件建立自动化验证。
- 需要把测试脚本接入 CI/CD，形成质量门禁。
- AI 生成代码后，需要补齐自动化测试验证正确性。

## 测试目标
- **验证业务行为**: 测试用户可感知的结果，而不是只测试实现细节。
- **覆盖关键风险**: 优先覆盖核心链路、权限、数据一致性、异常流程和边界条件。
- **可重复执行**: 测试应稳定、独立、可在本地和 CI 中重复运行。
- **可维护**: 测试命名清晰、数据准备可读，避免过度 mock 和脆弱断言。
- **反馈快速**: 单元测试快，集成/E2E 分层运行，避免所有测试都变慢。

## 测试分层策略

| 测试类型 | 重点 | 常用工具 |
|----------|------|----------|
| **单元测试** | 工具函数、业务规则、组件、Hook、Service 方法 | Jest、Vitest、JUnit、pytest、Go test |
| **组件测试** | UI 渲染、交互、表单校验、状态切换 | React Testing Library、Vue Test Utils |
| **接口测试** | 参数校验、响应结构、错误码、权限、幂等 | Supertest、pytest + requests、Postman/Newman |
| **集成测试** | API + DB + Cache + Queue 的真实协作 | Testcontainers、Docker Compose、Spring Test |
| **E2E 测试** | 登录、下单、支付、审批等完整用户路径 | Playwright、Cypress |
| **回归测试** | 已修复 bug 的复现和防回退 | 对应技术栈测试工具 |
| **服务端压力测试** | 响应时间、吞吐量、并发、错误率、资源占用 | k6、JMeter、Locust |

## 编写测试前需要收集的信息
- **需求与验收标准**: 这段功能应该产生什么业务结果。
- **测试对象边界**: 要测函数、组件、接口、服务还是完整用户流程。
- **关键路径**: 用户或系统最重要的成功流程。
- **异常路径**: 参数错误、权限不足、数据不存在、第三方失败、并发冲突。
- **测试数据**: 需要哪些用户、角色、数据库记录、文件、Mock 响应。
- **依赖服务**: 数据库、缓存、消息队列、对象存储、第三方 API 是否需要 mock 或容器化。
- **运行方式**: 本地命令、CI 命令、覆盖率要求、是否需要并行。

## 编写原则
- **先测行为，再测实现**: 优先断言输出、状态变化、接口响应和用户界面，而不是内部函数调用次数。
- **一个测试一个意图**: 测试名清楚表达场景和期望结果。
- **Arrange / Act / Assert**: 准备数据、执行动作、断言结果三段清晰。
- **测试数据最小化**: 只创建当前场景需要的数据，避免共享大 fixture。
- **避免测试互相依赖**: 每个测试可单独运行，执行顺序不影响结果。
- **Mock 有边界**: 单元测试可 mock 外部依赖；集成测试应尽量使用真实数据库或容器。
- **失败信息可定位**: 断言要具体，测试失败时能快速知道哪个业务条件不满足。

## 通用测试清单

### 1. 正常流程
- 是否覆盖最核心的 happy path？
- 是否断言最终业务结果，而不是只断言接口被调用？
- 是否验证关键字段、状态流转、页面展示或数据库写入？

### 2. 参数与边界
- 是否覆盖空值、缺失字段、非法类型、超长输入、边界数值？
- 是否覆盖分页、排序、过滤、时间范围、枚举值？
- 是否验证错误码、错误消息和 HTTP 状态码？

### 3. 权限与安全
- 是否覆盖未登录、无权限、跨用户、跨租户访问？
- 是否覆盖敏感操作的角色限制？
- 是否验证用户只能读写自己的资源？

### 4. 异常与失败
- 是否覆盖数据库无记录、第三方接口失败、超时、重复提交？
- 是否验证失败后不会产生脏数据？
- 是否验证错误提示和日志是否足够排查？

### 5. 幂等与并发
- 是否覆盖重复请求、重复回调、重复消费？
- 是否覆盖并发更新、库存扣减、状态流转冲突？
- 是否验证唯一约束、锁或幂等 key 生效？

### 6. 前端交互
- 是否覆盖 loading、empty、error、success 状态？
- 是否覆盖表单默认值、校验、提交中禁用、重复提交？
- 是否覆盖弹窗、抽屉、二次确认、路由跳转？

### 7. 回归测试
- Bug 修复是否补充“修复前会失败、修复后通过”的测试？
- 是否检查同类问题在其他入口是否也存在？
- 是否把历史高风险路径纳入稳定测试集？

### 8. 服务端压力测试
- 是否明确压测目标：接口 p95/p99、QPS/TPS、并发用户数、错误率、资源水位？
- 是否覆盖核心读写链路，而不是只压健康检查或静态接口？
- 是否准备了接近真实业务的测试数据、登录态、请求参数和请求比例？
- 是否区分基准测试、负载测试、压力测试、峰值测试、稳定性测试？
- 是否监控服务端 CPU、内存、GC、线程/协程、连接池、数据库慢查询、缓存命中率？
- 是否设置了合理的 ramp-up、持续时间和阈值，避免瞬时冲击导致误判？
- 是否避免压测生产环境；如必须压生产，是否有隔离、限流、告警、回滚和值班方案？
- 是否在压测后输出瓶颈分析、优化建议和复测结果？

## 服务端压力测试设计

### 压测类型

| 类型 | 目的 | 示例 |
|------|------|------|
| **基准测试** | 获取单接口在低并发下的基础耗时 | 1~5 VU，持续 1~5 分钟 |
| **负载测试** | 验证目标并发下是否满足 SLA | 100 VU，持续 10~30 分钟 |
| **压力测试** | 持续增加压力直到系统出现瓶颈 | 阶梯式升到 500/1000 VU |
| **峰值测试** | 模拟短时间流量尖峰 | 1 分钟内快速升高并发 |
| **稳定性测试** | 验证长时间运行是否资源泄漏 | 2~8 小时稳定负载 |

### 核心指标
- **吞吐量**: QPS、TPS、每分钟处理请求数。
- **延迟**: avg、p50、p90、p95、p99、max。
- **错误率**: HTTP 5xx、业务错误码、超时、连接失败。
- **资源使用**: CPU、内存、GC、线程池、连接池、FD、磁盘 IO、网络 IO。
- **依赖瓶颈**: 数据库慢查询、锁等待、缓存命中率、消息队列堆积、第三方接口耗时。
- **容量结论**: 在错误率和延迟满足 SLA 的前提下，系统可承载的最大 QPS/并发。

### 压测脚本编写原则
- 使用真实业务比例组织场景，例如登录 5%、列表查询 60%、详情 25%、写入 10%。
- 登录、鉴权、签名、租户、角色等上下文必须接近真实请求。
- 参数要随机化，避免缓存命中过高或只访问同一条数据。
- 写操作需要准备幂等 key 或测试隔离数据，压测后可清理。
- 为每个关键接口设置独立 threshold，不能只看整体平均耗时。
- 压测脚本中不要硬编码生产密钥、Token、手机号、身份证等敏感信息。

### k6 脚本示例

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 50 },
    { duration: '5m', target: 50 },
    { duration: '2m', target: 100 },
    { duration: '5m', target: 100 },
    { duration: '2m', target: 0 }
  ],
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<500', 'p(99)<1000']
  }
};

const BASE_URL = __ENV.BASE_URL || 'http://localhost:3000';
const TOKEN = __ENV.TEST_TOKEN;

export default function () {
  const headers = {
    Authorization: `Bearer ${TOKEN}`,
    'Content-Type': 'application/json'
  };

  const listRes = http.get(`${BASE_URL}/api/items?page=1&pageSize=20`, { headers });
  check(listRes, {
    'list status is 200': (r) => r.status === 200,
    'list has data': (r) => Boolean(r.json('data'))
  });

  sleep(1);
}
```

运行示例：

```bash
BASE_URL=http://localhost:3000 TEST_TOKEN=xxx k6 run scripts/load/items.k6.js
```

## CI 集成建议
- PR 阶段运行 lint、type check、单元测试和关键接口测试。
- 主干合并后运行完整集成测试和 E2E 冒烟测试。
- 发布前运行核心 E2E、数据库迁移验证、权限回归测试。
- 服务端重大改动、核心接口改动或容量相关改动后，在测试/预发环境运行压力测试并归档报告。
- 覆盖率只作为参考，核心路径覆盖比盲目追求百分比更重要。
- 测试失败应阻断合并或发布， flaky 测试必须尽快隔离和修复。

## 常见反模式
- 只测快照，不验证真实业务行为。
- 为了让测试通过而 mock 掉所有关键逻辑。
- 测试依赖执行顺序或共享全局状态。
- 使用固定 `sleep` 等待异步结果。
- 只覆盖 happy path，不覆盖权限、异常和边界。
- 断言过于宽松，例如只判断接口返回 200。
- 测试数据过大、过复杂，导致维护困难。
- 压测只看平均响应时间，不看 p95/p99、错误率和资源瓶颈。
- 压测脚本硬编码生产 Token、密钥或真实用户隐私数据。
- 未准备隔离数据就压写接口，导致测试污染业务数据。

## 能力提升建议
- 从真实线上 bug 中沉淀回归测试。
- 为每个核心业务模块维护测试场景清单。
- 学会阅读覆盖率报告，但不要被覆盖率绑架。
- 熟悉项目测试框架、Mock 工具、测试数据库和 CI 流程。
- 每次重构前先补行为测试，重构后用测试验证外部行为不变。
