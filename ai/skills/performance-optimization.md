# 性能优化 Skill

这个 Skill 用于指导 AI 或工程师识别、评估和优化系统各层面的性能问题，兼顾可用性、稳定性和成本，避免单纯追求微小性能提升而牺牲安全性和可维护性。

## 适用场景
- 新功能或改造上线前的性能评审。
- 线上性能回归、慢请求、CPU/MEM 升高的排查。
- 大列表、大文件、大数据量场景的优化方案评估。
- 缓存、数据库、网络、前端渲染等性能专项检查。
- 性能优化改造后的验证与回归测试评审。

## 性能优化目标
- **感知性能**: 提升用户可观察到的响应速度、首屏时间、交互流畅度。
- **系统性能**: 降低请求延迟、CPU/内存占用、数据库负载、网络带宽。
- **规模性能**: 确保系统在高并发、大数据量下保持稳定并可扩展。
- **成本效率**: 在可接受的体验范围内，避免过度资源消耗和复杂度。
- **稳定性**: 优化后系统应保持功能一致、错误可控、监控可观测。

## 优化原则
- **先测量后优化**: 先收集数据，找到真实瓶颈，而不是凭感觉改代码。
- **优化有边界**: 仅在业务关键路径和明显瓶颈处优化，不要过度优化冷门代码。
- **保持行为一致**: 不能为性能牺牲正确性、兼容性或安全性。
- **渐进验证**: 每次优化后都要验证效果，避免引入回归或稳定性问题。
- **优先级原则**: 先解决大影响、低成本的问题，再考虑复杂方案。

## 性能优化检查清单

### 1. 数据库与存储
- 是否存在未加索引的高频查询、全表扫描、范围扫描？
- 是否出现 N+1 查询或重复查询？
- 是否返回了过多列、过多行、未分页的数据？
- 写操作是否频繁、是否可以批量写入或异步处理？
- 是否合理使用缓存、并考虑缓存失效、更新策略和一致性？
- 是否有慢查询、锁等待、死锁、长事务、表扫描、索引失效？

### 2. 后端服务
- 是否存在重复计算、重复请求、无效调用？
- 是否对热点数据、热点接口做了限流、降级或熔断？
- 是否有不必要的同步阻塞、串行依赖或单线程瓶颈？
- 是否合理设置超时、重试、连接池和资源限制？
- 是否有大对象/大数组复制、深度序列化、低效排序、正则、字符串拼接？
- 是否存在过长请求路径、跨服务调用过多、请求链路复杂？

### 3. 前端体验
- 是否有不必要的全量渲染、重复渲染、无效 DOM 更新？
- 是否存在过多或重复的网络请求？
- 是否合理使用懒加载、按需加载、缓存、预加载？
- 是否考虑交互优先渲染、骨架屏、分片渲染和耗时任务拆分？
- 是否避免阻塞主线程的同步计算、长任务、过度动画？
- 是否按需加载依赖、减少首屏 JS/CSS 体积？

### 4. 网络与传输
- 是否请求了不必要的数据，或未启用压缩、缓存策略？
- 是否避免了大文件/大图片/大 JSON 的不必要传输？
- 是否合并请求、批量请求或减少请求次数？
- 是否度量了网络耗时、DNS、TLS、首字节时间、下载时间？
- 是否考虑 CDN、边缘缓存、静态资源缓存策略？

### 5. 计算与资源
- 是否出现内存泄漏、对象保留、无界队列或无限增长结构？
- 是否合理控制并发、线程池、协程数量、连接数？
- 是否有 GC 压力、频繁分配大对象、缓冲区重分配？
- 是否使用了合适的数据结构与算法？
- 是否避免了过度缓存、缓存雪崩、热缓存失效事故？

## 验证与回归
- 是否提供优化前后的性能数据对比？
- 是否有稳定性指标，如错误率、成功率、资源占用变动？
- 是否补充了回归测试、压力测试、自动化验证？
- 是否记录了优化方案、风险点、变更范围和回滚方案？

## 风险提示
- 缓存优化要注意一致性、缓存穿透、缓存雪崩。
- 数据库索引优化要注意写扩散、空间成本和排序字段匹配。
- 前端引入过多优化方案可能增加复杂度、降低可维护性。
- 性能改动后要关注监控、日志、异常、用户体验是否异常。

## 推荐输出结构
- **现状**: 当前关键性能问题、指标与复现方式。
- **原因分析**: 瓶颈位置、根因、业务影响。
- **优化方案**: 具体改动方案、优先级、成本评估。
- **验证结果**: 优化前后数据、验证方法、稳定性观察。
- **风险与落地建议**: 兼容性、监控、回滚、后续关注点。
