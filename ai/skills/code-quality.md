# 代码质量评估 Skill

这个 Skill 用于指导 AI 或工程师评估代码质量，重点关注 lint 规范、重复代码、圈复杂度、设计模式选用和模块拆分是否合理，确保代码既易维护又符合工程规范。

## 适用场景
- 代码提交、PR/ MR 审查时的质量评估。
- 代码重构、模块拆分、架构优化前的质量检查。
- 代码质量专项巡查，例如技术债务、重复代码、复杂度过高场景。
- 开发规范落地和团队代码一致性评估。

## 代码质量目标
- **代码规范**: 通过 lint 检查确保一致性、可读性和规则覆盖。
- **重复度**: 识别重复逻辑、重复实现，减少冗余代码。
- **复杂度**: 控制圈复杂度、认知负担和代码路径数量。
- **设计合理性**: 验证设计模式和架构是否适配当前业务。
- **模块化**: 确保拆分和依赖关系清晰、职责单一、可复用性高。

## 评估原则
- **先分析再结论**: 先定位问题点，再给出具体建议，不直接照抄规则。
- **实用优先**: 关注对团队可维护性、可扩展性和错误率影响最大的质量项。
- **整体视角**: 既看单文件质量，也看模块边界、依赖关系和设计一致性。
- **轻重分级**: 将“必须修复”和“建议优化”区分开，避免过度追求完美。
- **业务适配**: 不强制过度设计，设计模式的选用应服务于业务复杂度和扩展需求。

## 质量评估清单

### 1. lint 与代码规范
- 是否遵循项目 lint 规则（ESLint、Prettier、Stylelint 等）？
- 是否存在命名不规范、未使用变量、空语句、magic number、未处理 Promise 等问题？
- 是否有 ts/ js 类型检查异常、隐式 any、非空断言滥用等问题？
- 是否使用了一致的缩进、分号、引号和空行规则？
- 是否有不必要的注释、死代码、console.log 或调试代码残留？

### 2. 重复代码与抽象
- 是否存在相似逻辑在多个文件/方法中重复出现？
- 是否有可提取为公共工具函数、基础组件或中间件的重复逻辑？
- 是否存在“半重复”逻辑，复制粘贴后又做小改动，导致维护风险？
- 是否避免了过度抽象：抽象方式应简单明了，不要为去重而制造复杂层级。

### 3. 圈复杂度与可读性
- 是否存在长函数、长方法、深层嵌套、复杂条件分支？
- 是否能把复杂逻辑拆分成更小的单一职责函数/组件？
- 是否有大量嵌套回调、状态机条件分支、switch-case 或 if-else 链条？
- 是否使用了适当的早期返回、guard clause、单向数据流来降低复杂度？
- 是否存在“feature envy”或“不属于当前模块/函数职责”的逻辑？

### 4. 设计模式与架构选型
- 当前的设计模式是否适配当前业务和复杂度？
- 是否存在过度设计，例如简单需求使用过重的抽象、策略、工厂、装饰器等模式？
- 是否存在设计不合理，例如“神对象”、“泛化过头”的基础类、多个职责混杂的服务？
- 是否考虑了扩展性、可测试性和耦合度，避免紧耦合实现？
- 是否有更简单、可维护的替代方案，如组合优于继承、纯函数优先、领域模型简化？

### 5. 模块拆分与依赖关系
- 当前模块划分是否满足单一职责原则？
- 是否存在模块职责不清、功能散落、多模块循环依赖、扁平耦合过高？
- 是否按照领域或边界划分模块，而不是随意按文件夹堆砌？
- 是否有模块内部逻辑过多、导出太多公共接口、对外暴露实现细节？
- 是否考虑了模块复用、可替换性和边界清晰度？

## 输出建议结构
- **质量摘要**: 代码质量现状、最关键的问题类型和风险。
- **问题列表**: 具体发现的 lint 规则、重复逻辑、复杂度热点、设计/拆分问题。
- **影响评估**: 为什么这些问题会影响可维护性、开发效率或上线风险。
- **改进建议**: 具体改进思路、重构方向、规则加固、模块边界调整建议。
- **验证方式**: 可执行的质量验证准则，如 lint 通过、复杂度下降、代码走查覆盖。
