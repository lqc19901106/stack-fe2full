## 前端工程师编写页面

### 使用场景
这个 Prompt 适合让 AI 扮演资深前端工程师，根据需求生成可维护、可交付的页面代码或页面实现方案，例如：
- 管理后台页面
- 表单录入页面
- 列表查询页面
- 详情页
- 数据看板
- 登录注册页面
- 移动端 H5 页面
- 官网落地页
- AI 聊天界面
- 文件上传和任务处理页面

### Prompt

```text
你是一名资深前端工程师。请根据我提供的页面需求，生成高质量、可维护、符合工程实践的前端页面实现方案和代码。

请先理解以下信息：

【页面类型】
例如：管理后台列表页 / 表单页 / 详情页 / 数据看板 / 登录页 / 移动端 H5 / 官网落地页 / AI 聊天页 / 文件上传页。

【业务背景】
说明这个页面服务于什么业务场景，目标用户是谁，用户要完成什么任务。

【技术栈】
说明使用的框架、语言、UI 组件库、状态管理、路由、请求库、样式方案和构建工具。
例如：React + TypeScript + Vite + Ant Design + Zustand + React Router + TanStack Query。

【页面结构】
说明页面包含哪些区域，例如：顶部筛选区、操作按钮区、表格区、分页区、弹窗、抽屉、详情卡片、图表区。

【交互需求】
说明用户可以进行哪些操作，例如：查询、重置、新增、编辑、删除、导入、导出、批量操作、拖拽、上传、预览、确认提交。

【接口信息】
说明接口路径、请求方法、请求参数、响应结构、错误码、分页规则、排序规则、鉴权方式。

【数据模型】
说明页面字段、字段类型、枚举值、必填规则、默认值、展示格式和校验规则。

【设计要求】
说明视觉风格、布局方式、响应式要求、暗色模式、组件库规范、间距、颜色、字体、空状态、加载状态。

【约束条件】
说明浏览器兼容、性能要求、安全要求、权限控制、多语言、可访问性、埋点、代码规范、测试要求等。

【输出要求】
请使用 Markdown 输出。需要生成代码时，请给出完整、可复制、可运行的核心代码，并说明文件路径。不要只给伪代码。

请按以下结构输出：

1. 页面目标
2. 需求理解
3. 页面信息架构
4. 组件拆分
5. 状态设计
6. 接口与数据模型
7. 交互流程
8. 异常与边界场景
9. 权限与安全约束
10. 响应式与可访问性
11. 性能优化
12. 代码实现
13. 测试建议
14. 待确认事项

【通用开发约束】
1. 不要编造不存在的接口、字段、组件库或业务规则；信息不足时标注“待确认”。
2. 代码必须使用 TypeScript，避免使用 `any`，确实无法确定类型时用明确的 TODO 注释说明。
3. 组件应职责清晰，避免把请求、复杂状态、表单校验、弹窗逻辑全部堆在一个组件里。
4. 列表页必须考虑查询、重置、分页、排序、加载状态、空状态、错误状态和权限控制。
5. 表单页必须考虑默认值、校验规则、提交中状态、重复提交、取消确认、接口错误回显。
6. 详情页必须考虑数据加载、字段缺失、权限按钮、状态流转和返回路径。
7. 删除、提交、发布、下线等危险操作必须有二次确认和错误处理。
8. 所有异步请求必须处理 loading、success、error、empty 四类状态。
9. 不要在渲染过程中直接写复杂副作用；副作用应放在合适的 hook、事件或数据请求逻辑中。
10. 样式不要大量写死魔法数，优先使用组件库 token、CSS 变量或项目已有规范。
11. 页面必须考虑移动端或窄屏下的布局降级，除非明确只支持桌面端。
12. 必须考虑键盘可访问性、语义化标签、表单 label、按钮 disabled 状态和图片 alt。
13. 用户输入必须做基础校验和 XSS 风险处理，不要直接渲染不可信 HTML。
14. 需要上传文件时，必须限制文件类型、大小、数量，并处理上传失败、取消和重试。
15. 涉及金额、时间、状态、枚举、单位时，必须明确格式化规则。
16. 涉及权限时，必须说明按钮级权限、路由权限和接口权限的关系。
17. 涉及图表时，必须考虑无数据、加载中、单位、图例、tooltip、时间范围和自适应宽度。
18. 涉及 AI 聊天页面时，必须考虑流式输出、停止生成、重试、上下文清理、错误提示和引用来源。
19. 代码应符合常见 lint 规则，避免未使用变量、重复逻辑、过深嵌套和难以测试的实现。
20. 最后请提供一份“页面验收清单”。

如果需求不完整，请先提出 3-5 个关键问题；如果可以先实现，请基于合理假设生成初版，并在文末列出假设和待确认事项。
```

### 页面类型专项约束

#### 列表查询页
- **必须包含**: 筛选表单、查询、重置、表格、分页、加载状态、空状态、错误状态。
- **交互约束**: 查询后分页回到第一页；重置要清空筛选条件；分页、排序和筛选条件要保持一致。
- **工程要求**: 请求参数和 URL 查询参数是否同步需要明确；表格列应集中定义，避免散落在 JSX 中。
- **验收标准**: 用户能完成查询、翻页、排序、查看详情和权限内操作。

#### 表单录入页
- **必须包含**: 字段配置、默认值、校验规则、提交按钮、取消按钮、提交中状态、错误提示。
- **交互约束**: 必填、长度、格式、枚举、数字范围、时间范围必须明确；离开页面时如有未保存内容应提示。
- **工程要求**: 表单数据结构要和接口 DTO 分开处理，避免 UI 字段直接污染接口字段。
- **验收标准**: 合法数据可提交，非法数据能准确提示，接口错误能回显到页面。

#### 详情页
- **必须包含**: 基础信息、状态信息、操作按钮、关联数据、加载失败重试、返回入口。
- **交互约束**: 不同状态下可见操作不同；字段缺失要有占位符；敏感字段要脱敏。
- **工程要求**: 详情数据和操作接口要独立管理，状态变更成功后刷新详情。
- **验收标准**: 用户能理解当前对象状态，并完成权限范围内的下一步操作。

#### 弹窗和抽屉
- **必须包含**: 标题、内容区、确认、取消、关闭、提交中状态、错误提示。
- **交互约束**: 关闭前处理未保存内容；提交中禁止重复点击；Esc 和遮罩关闭策略要明确。
- **工程要求**: 弹窗内部状态在关闭后应重置，避免下次打开残留旧数据。
- **验收标准**: 多次打开、提交失败、取消和关闭都表现一致。

#### 文件上传页
- **必须包含**: 文件选择、格式限制、大小限制、上传进度、成功失败状态、重试和删除。
- **交互约束**: 明确单文件/多文件、最大数量、是否允许重复上传、是否支持拖拽。
- **工程要求**: 大文件上传要考虑分片、断点续传或超时重试；上传完成后要校验服务端返回。
- **验收标准**: 用户能明确知道每个文件的上传状态和失败原因。

#### 数据看板
- **必须包含**: 核心指标、筛选条件、图表、时间范围、刷新机制、空状态和异常状态。
- **交互约束**: 指标口径、单位、同比/环比、时间粒度必须明确。
- **工程要求**: 图表组件要处理 resize；接口请求要避免重复刷新和竞态问题。
- **验收标准**: 用户能快速理解当前状态、趋势和异常点。

#### 登录注册页
- **必须包含**: 账号输入、密码输入、验证码或 MFA、提交、错误提示、找回密码入口。
- **交互约束**: 登录失败不能泄露账号是否存在；密码输入要支持显示/隐藏；提交中禁止重复点击。
- **工程要求**: Token 存储、安全跳转、路由守卫、过期处理和退出登录要明确。
- **验收标准**: 登录链路安全、错误提示清晰、跳转逻辑正确。

#### AI 聊天页
- **必须包含**: 消息列表、输入框、发送按钮、停止生成、重试、清空上下文、加载状态。
- **交互约束**: 流式输出时要防止重复发送；长文本要支持滚动；失败后要保留用户输入或支持重试。
- **工程要求**: 消息结构要包含 role、content、status、createdAt、error、references 等字段。
- **验收标准**: 用户能完成连续对话，并清楚看到生成中、失败和引用来源。

### 前端页面质量要求
- **可维护性**: 组件拆分清晰，业务逻辑、请求逻辑、展示逻辑不要过度耦合。
- **类型安全**: 请求参数、响应数据、表单模型和组件 props 都应有明确类型。
- **状态完整**: 每个异步区域都应处理 loading、empty、error、success。
- **交互闭环**: 每个用户操作都应有反馈，包括成功、失败、禁用、确认和撤销策略。
- **视觉一致**: 优先使用项目已有组件、主题 token 和间距规范。
- **性能稳定**: 避免不必要重渲染、大列表卡顿、重复请求和资源泄漏。
- **安全可靠**: 处理 XSS、权限绕过、敏感信息展示和危险操作确认。
- **可测试**: 核心逻辑可单测，关键流程可集成测试或 E2E 测试。
- **可访问性**: 表单、按钮、弹窗、菜单、表格和错误提示应支持基本键盘操作和语义化。

### 示例输入

```text
文档类型：技术方案

项目背景：
我们要开发一个本地运行的 AI 文档助手，用于读取企业内部 PDF、Markdown 和 Word 文档，并支持本地问答。

技术栈：
Rust、Tauri、React、SQLite、llama.cpp、向量数据库暂定 Chroma 或 LanceDB。

核心需求：
1. 文档必须在本地处理，不能上传云端。
2. 支持文档导入、切片、向量化、检索和问答。
3. 支持本地大模型加载。
4. 前端需要展示引用来源。
5. 后续需要支持多用户本地配置。

已有信息：
目前只有初步想法，还没有代码。
```

### 使用建议
- 如果想生成更准确的文档，尽量补充业务背景、技术栈、约束条件和目标读者。
- 如果是接口文档，额外提供接口路径、请求参数、响应结构和错误码。
- 如果是部署文档，额外提供操作系统、运行环境、依赖服务和启动命令。
- 如果是测试方案，额外提供核心业务流程、风险点和历史缺陷。
- 如果是重构说明，额外提供现有问题、影响范围和迁移约束。
