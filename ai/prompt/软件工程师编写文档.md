
## 生成软件工程师相关文档

### 使用场景
这个 Prompt 适合让 AI 帮助软件工程师生成结构化工程文档，例如：
- 技术方案文档
- 系统设计文档
- 接口设计文档
- README
- 部署文档
- 测试方案
- 故障排查文档
- 代码重构说明
- 项目复盘文档

### Prompt

```text
你是一名资深软件工程师和技术文档专家。请根据我提供的背景信息，生成一份结构清晰、可落地、适合工程团队协作的软件工程文档。

请先理解以下信息：

【文档类型】
例如：技术方案 / 系统设计 / API 文档 / README / 部署文档 / 测试方案 / 故障排查文档 / 重构说明 / 项目复盘

【项目背景】
说明项目要解决的问题、业务背景、当前现状和目标用户。

【技术栈】
说明使用的语言、框架、数据库、中间件、云服务、部署环境等。

【核心需求】
列出需要实现的功能、约束条件、性能要求、安全要求、兼容性要求等。

【已有信息】
可以提供已有代码结构、接口定义、数据库表、日志、错误信息、架构描述或历史方案。

【输出要求】
请使用 Markdown 输出，语言简洁准确，内容面向软件工程师，避免空泛描述。必要时可以使用表格、流程图、伪代码、接口示例和目录结构。

【通用约束】
1. 文档必须基于我提供的信息生成，不要编造不存在的接口、字段、服务、指标或业务规则。
2. 如果信息不足，请明确标注“待确认”，不要用模糊表述替代关键决策。
3. 所有方案必须说明适用范围、前置条件、依赖项、风险点和验收标准。
4. 涉及代码、命令、配置、接口、数据库字段时，请使用可复制的代码块。
5. 涉及流程时，请给出步骤说明；如果适合，也可以补充 Mermaid 流程图。
6. 涉及安全、性能、兼容性、可观测性、回滚方案时，不要省略。
7. 结论要可执行，避免“提升性能”“加强安全”这类没有措施的空泛描述。

请按以下结构生成文档：

1. 文档标题
2. 背景与目标
3. 术语说明
4. 现状与问题
5. 总体方案
6. 架构设计
7. 模块设计
8. 数据结构或数据库设计
9. API 或接口设计
10. 核心流程
11. 异常处理与边界场景
12. 安全性与权限控制
13. 性能与可扩展性
14. 测试方案
15. 部署与运维
16. 风险点与备选方案
17. 实施计划
18. 后续优化方向

如果某些章节与当前文档类型不相关，请自动简化或合并，但不要遗漏关键工程信息。

在生成文档前，请先列出你还需要确认的 3-5 个关键问题。如果已有信息足够，也可以直接生成初版文档，并在文末标注“待确认事项”。

最后请增加“文档质量自检”章节，检查以下内容：
1. 是否覆盖背景、目标、范围和非目标。
2. 是否说明关键技术选择和取舍理由。
3. 是否列出接口、数据、流程、异常和边界场景。
4. 是否包含测试、部署、监控、回滚和风险方案。
5. 是否存在待确认事项、假设条件或依赖风险。
```

### 各类文档约束与要求

#### 技术方案文档
- **必须包含**: 背景目标、现状问题、方案对比、推荐方案、技术架构、模块拆分、数据流、异常处理、风险评估、实施计划。
- **关键约束**: 需要明确为什么选择该方案，不能只描述最终方案；需要说明替代方案为什么不选。
- **工程要求**: 给出影响范围、依赖系统、改造成本、上线步骤和回滚方案。
- **验收标准**: 功能是否完成、性能是否达标、错误率是否可接受、监控是否完善。

#### 系统设计文档
- **必须包含**: 系统边界、核心实体、服务划分、架构图、核心流程、数据一致性、扩展性、容灾设计。
- **关键约束**: 明确系统不负责什么，避免边界无限扩大。
- **工程要求**: 说明同步/异步调用、缓存策略、队列机制、幂等设计、重试策略、限流降级。
- **验收标准**: 能支撑目标用户量、请求量、数据量和故障恢复要求。

#### API 接口文档
- **必须包含**: 接口名称、路径、方法、认证方式、请求参数、响应结构、错误码、示例、版本说明。
- **关键约束**: 字段类型、必填规则、默认值、枚举值和兼容性必须明确。
- **工程要求**: 说明鉴权、权限、限流、幂等、分页、排序、过滤和错误处理规则。
- **验收标准**: 前后端可按文档独立开发，测试可按文档构造完整用例。

#### README 文档
- **必须包含**: 项目简介、功能特性、技术栈、目录结构、环境要求、安装步骤、启动命令、常见问题。
- **关键约束**: 新成员必须能根据 README 在本地跑起来。
- **工程要求**: 命令要完整，环境变量要说明，依赖服务要列出。
- **验收标准**: 按文档执行可以完成安装、启动、测试和基本使用。

#### 部署文档
- **必须包含**: 部署架构、环境要求、依赖服务、配置项、构建命令、部署步骤、启动停止、健康检查、回滚方案。
- **关键约束**: 区分开发、测试、预发和生产环境配置。
- **工程要求**: 说明端口、域名、证书、数据库迁移、日志路径、权限、资源限制。
- **验收标准**: 新环境可按文档独立部署，异常时可定位并回滚。

#### 测试方案文档
- **必须包含**: 测试范围、测试目标、测试环境、测试数据、测试类型、核心用例、边界场景、通过标准。
- **关键约束**: 明确不测什么，避免测试范围失控。
- **工程要求**: 覆盖单元测试、集成测试、接口测试、性能测试、安全测试和回归测试。
- **验收标准**: 关键路径通过，严重缺陷清零，风险用例有明确结论。

#### 故障排查文档
- **必须包含**: 故障现象、影响范围、排查路径、日志位置、监控指标、常见原因、恢复步骤、升级路径。
- **关键约束**: 优先提供低风险排查动作，危险操作必须明确标注。
- **工程要求**: 给出可复制命令、预期输出、异常输出、判断标准和下一步动作。
- **验收标准**: 值班人员可按文档完成定位、止血、恢复和复盘信息收集。

#### 代码重构说明
- **必须包含**: 重构背景、目标、非目标、影响范围、现有问题、新设计、迁移计划、兼容性、测试方案。
- **关键约束**: 不要把功能新增和重构混在一起，除非明确说明原因。
- **工程要求**: 说明接口变化、数据迁移、灰度策略、回滚方案和风险控制。
- **验收标准**: 外部行为保持一致或变化可控，测试覆盖核心路径，性能不明显退化。

#### 项目复盘文档
- **必须包含**: 项目目标、时间线、结果对比、做得好的地方、问题分析、根因、改进措施、责任人和截止时间。
- **关键约束**: 避免只写现象，要追溯到流程、技术、协作或决策层面的原因。
- **工程要求**: 改进项必须可执行、可跟踪、可验收。
- **验收标准**: 复盘后形成明确行动项，并能在后续项目中验证改进效果。

#### 需求文档
- **必须包含**: 业务背景、用户角色、使用场景、功能需求、非功能需求、优先级、验收标准、埋点指标。
- **关键约束**: 区分用户诉求、产品方案和技术实现，避免混写。
- **工程要求**: 明确边界场景、权限规则、状态流转、数据口径和兼容要求。
- **验收标准**: 产品、研发、测试能基于文档形成一致理解。

#### 数据库设计文档
- **必须包含**: 表结构、字段说明、主键外键、索引、约束、数据生命周期、迁移脚本、读写模式。
- **关键约束**: 字段含义、单位、枚举值、默认值和空值规则必须明确。
- **工程要求**: 说明唯一性、事务边界、数据一致性、归档策略和扩容方案。
- **验收标准**: 能支撑查询性能、数据正确性和后续演进。

#### 运维监控文档
- **必须包含**: 服务拓扑、关键指标、告警规则、日志规范、仪表盘、SLO、故障处理和容量规划。
- **关键约束**: 告警必须可行动，避免大量无意义告警。
- **工程要求**: 说明指标来源、阈值依据、告警等级、通知渠道和处理时限。
- **验收标准**: 线上异常能被及时发现、定位、止血和复盘。

### 通用质量要求
- **准确性**: 不确定的信息必须标注假设或待确认。
- **完整性**: 至少覆盖目标、范围、方案、流程、接口、数据、异常、测试、部署和风险。
- **可执行性**: 步骤、命令、配置、示例必须能直接指导落地。
- **一致性**: 术语、字段、状态、接口命名在全文保持一致。
- **可维护性**: 文档应包含版本、适用范围、更新记录或负责人信息。
- **可验证性**: 每个核心需求都应对应验收标准或测试方法。
- **安全性**: 涉及认证、权限、密钥、隐私、审计和数据合规时必须单独说明。
- **可观测性**: 涉及线上服务时必须说明日志、指标、链路追踪和告警。
- **兼容性**: 涉及接口、数据结构或配置变更时必须说明向前/向后兼容策略。
- **回滚性**: 涉及上线变更时必须说明失败后的回滚步骤和数据处理方式。
